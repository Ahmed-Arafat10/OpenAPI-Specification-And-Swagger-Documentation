## Role of Data Types in OpenAPI

In OpenAPI, **data types are always defined using the `type` keyword**.

They are primarily used inside:

* `schema` objects
* request bodies
* responses
* parameters
* components

Every request and response structure ultimately breaks down into these data types.

## Supported OpenAPI Data Types (High Level)

OpenAPI officially supports the following core types:

| Type    | Purpose                                       |
|---------|-----------------------------------------------|
| string  | Textual data, dates, date-time, binary, enums |
| integer | Whole numbers                                 |
| number  | Decimal / floating-point numbers              |
| boolean | true / false values                           |
| object  | Structured key-value data                     |
| array   | Lists of values or objects                    |

Important note:

> **OpenAPI does NOT introduce custom primitive data types**
> Everything builds on top of these core ones.

## YAML Examples — Core Data Types

### 1️⃣ String

```yaml
schema:
  type: string
```

Used for:

* names
* descriptions
* identifiers
* textual values

---

### 2️⃣ Integer

```yaml
schema:
  type: integer
```

Used for:

* IDs
* counts
* quantities
* pagination values

---

### 3️⃣ Number

```yaml
schema:
  type: number
```

Used for:

* prices
* ratings
* measurements
* decimal values

---

### 4️⃣ Boolean

```yaml
schema:
  type: boolean
```

Used for:

* flags
* feature toggles
* yes/no fields

Example in an object:

```yaml
type: object
properties:
  isActive:
    type: boolean
```

---

### 5️⃣ Object

```yaml
schema:
  type: object
  properties:
    id:
      type: integer
    name:
      type: string
```

Used when:

* grouping related fields
* defining structured data
* modeling domain entities

---

### 6️⃣ Array

```yaml
schema:
  type: array
  items:
    type: string
```

Important rules:

* `type: array` **must always have `items`**
* `items` defines the data type of elements inside the array

Example with objects:

```yaml
schema:
  type: array
  items:
    type: object
    properties:
      id:
        type: integer
      name:
        type: string
```

## Date, Time, and Special Formats

OpenAPI does **not** introduce separate `date` or `datetime` types.

Instead, these are represented using **`string` + `format`**.

Examples (preview – explained in later lectures):

```yaml
type: string
format: date
```

```yaml
type: string
format: date-time
```

```yaml
type: string
format: binary
```

These are still **strings**, but with semantic meaning.

## Why This Matters

Understanding data types deeply is critical because:

* Every schema is built from them
* Validation rules depend on them
* Code generation relies on them
* Mistakes here propagate everywhere

Later topics like:

* polymorphism
* inheritance
* oneOf / anyOf / allOf

**only make sense if data types are already clear**.
