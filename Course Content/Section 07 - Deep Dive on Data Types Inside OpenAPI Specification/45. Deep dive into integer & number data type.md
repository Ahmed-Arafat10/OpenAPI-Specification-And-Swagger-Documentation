## 45. Deep Dive into `integer` & `number` Data Types

In this lecture, the focus is on **how numeric values are represented** in the OpenAPI Specification and how you can **add precision and constraints** to numeric fields so that both humans and tools clearly understand your API contracts.

You have already been using numeric types throughout the course, but this lecture formalizes **best practices and advanced options**.

---

## Integer vs Number — Core Difference

OpenAPI provides **two numeric data types**:

### `integer`

* Used for **whole numbers only**
* **No decimal values allowed**
* Typical use cases:

    * IDs
    * counts
    * quantities
    * pagination values

### `number`

* Used for **decimal values**
* Supports floating-point numbers
* Typical use cases:

    * prices
    * measurements
    * ratings
    * percentages

---

## YAML Example — Basic Integer

```yaml
schema:
  type: integer
```

This explicitly tells consumers:

> This field accepts only whole numbers.

---

## YAML Example — Basic Number

```yaml
schema:
  type: number
```

This indicates:

> Decimal values are allowed (float or double).

---

## Using the `format` Keyword (Optional but Powerful)

Both `integer` and `number` support an **optional** keyword called `format`.

The `format` keyword:

* adds **semantic meaning**
* helps **code generators**
* improves **consumer clarity**
* improves **mock servers and validation**

---

## Integer Formats

| Format  | Meaning                                    |
| ------- | ------------------------------------------ |
| `int32` | 32-bit signed integer                      |
| `int64` | 64-bit signed integer (very large numbers) |

### YAML Example — `int32`

```yaml
schema:
  type: integer
  format: int32
```

### YAML Example — `int64`

```yaml
schema:
  type: integer
  format: int64
```

Use `int64` when:

* IDs are very large
* values may exceed normal integer limits

---

## Number Formats

| Format   | Meaning                  |
| -------- | ------------------------ |
| `float`  | Single-precision decimal |
| `double` | Double-precision decimal |

### YAML Example — Float

```yaml
schema:
  type: number
  format: float
```

### YAML Example — Double

```yaml
schema:
  type: number
  format: double
```

If `format` is **not provided**, the number:

* may be treated as float or double
* has **no precision guarantee**

---

## Practical Example — Product Price

Before using `format`:

```yaml
price:
  type: number
```

This is ambiguous:

* Is it float?
* Is it double?

After improvement:

```yaml
price:
  type: number
  format: float
```

This clearly communicates:

> The price supports decimal values with floating-point precision.

Swagger UI will now visually show:

```
number($float)
```

---

## Adding Value Constraints (`minimum` & `maximum`)

Both `integer` and `number` support **range constraints**.

These are optional but highly recommended for:

* validation
* business rules
* consumer clarity

---

### YAML Example — Integer with Range

```yaml
schema:
  type: integer
  minimum: 1
  maximum: 100
```

Meaning:

* Value must be between **1 and 100**
* Inclusive by default

---

### YAML Example — Number with Range

```yaml
schema:
  type: number
  format: float
  minimum: 0.0
  maximum: 9999.99
```

Used for:

* prices
* balances
* measurements

---

## Examples Must Match the Schema

A **very important rule** emphasized in this lecture:

> Example values must align with the declared data type and format.

### ❌ Bad Example

```yaml
price:
  type: number
  format: float
  example: 999
```

This looks like an integer and creates ambiguity.

### ✅ Good Example

```yaml
price:
  type: number
  format: float
  example: 999.25
```

This clearly shows:

* decimal support
* correct numeric intent

---

## YAML Example — Product Schema with Numbers

```yaml
Product:
  type: object
  properties:
    id:
      type: integer
      format: int64
    price:
      type: number
      format: float
      minimum: 0.0
    quantity:
      type: integer
      minimum: 1
```

This schema is:

* precise
* tool-friendly
* consumer-friendly
* production-ready

---

## Key Takeaways from This Lecture

* Use `integer` for **whole numbers only**
* Use `number` for **decimal values**
* Use `format` when precision matters
* Use `minimum` and `maximum` to enforce boundaries
* Ensure **examples always match schema definitions**
* Small schema improvements significantly improve API usability

This lecture strengthens your numeric modeling foundation, which is essential before moving into **advanced schema composition and validation techniques** later in the course.
