## 49. Deep Dive into **Array** Data Type

Arrays are one of the most frequently used data types in OpenAPI specifications. They allow you to model **lists of values** such as collections of products, orders, IDs, or even nested data structures.

In OpenAPI, arrays are always described using **two mandatory keywords together**:

* `type: array`
* `items`

---

## Basic Array Definition

Whenever you define an array, **`items` is mandatory**.
`items` explains **what each element inside the array looks like**.

### YAML Example â€” Array of Integers

```yaml
type: array
items:
  type: integer
```

Meaning:

* The value must be a list
* Every element in the list must be an integer

---

## Array of Objects (Most Common Case)

Arrays often contain **objects**, especially in responses.

### YAML Example â€” Array of Objects Using Inline Schema

```yaml
type: array
items:
  type: object
  properties:
    id:
      type: integer
    name:
      type: string
```

---

## Array Referencing a Schema from `components`

Best practice is to define object schemas once and reuse them.

### YAML Example â€” Schema Definition

```yaml
components:
  schemas:
    Product:
      type: object
      properties:
        id:
          type: integer
        name:
          type: string
        price:
          type: number
```

### YAML Example â€” Array Referencing That Schema

```yaml
type: array
items:
  $ref: '#/components/schemas/Product'
```

This avoids duplication and keeps your specification maintainable.

---

## Restricting Array Size (`minItems`, `maxItems`)

You can control how many elements an array can contain.

### `minItems`

* Prevents empty arrays
* Ensures at least N elements are present

### `maxItems`

* Limits the maximum number of elements

### YAML Example â€” Size Restrictions

```yaml
type: array
minItems: 1
maxItems: 10
items:
  type: string
```

Meaning:

* At least **1** item must exist
* At most **10** items allowed

---

## Enforcing Unique Elements (`uniqueItems`)

If duplicates are not allowed, you can enforce uniqueness.

### YAML Example â€” Unique Values Only

```yaml
type: array
uniqueItems: true
items:
  type: integer
```

Effect:

* `[1, 2, 3]` âś… valid
* `[1, 2, 2]` âťŚ invalid

This is especially useful for:

* IDs
* Codes
* Category lists
* Tags

---

## Nested Arrays (Array of Arrays)

OpenAPI allows **nested arrays**, although they should be used carefully to avoid complexity.

### YAML Example â€” Array of Arrays of Integers

```yaml
type: array
items:
  type: array
  items:
    type: integer
```

This represents:

```json
[
  [1, 2, 3],
  [4, 5],
  [6]
]
```

You can nest arrays to any depth, but in real-world APIs, **deep nesting is discouraged** unless absolutely required.

---

## Common Mistakes to Avoid

âťŚ Missing `items` when using `type: array`
âťŚ Using `minItems` or `maxItems` without `items`
âťŚ Forgetting `$ref` when reusing schemas
âťŚ Overusing nested arrays when objects are clearer

---

## Key Takeaways

* `items` is **mandatory** for arrays
* Arrays can contain:

    * Primitives (string, integer, number, boolean)
    * Objects
    * Other arrays
* Use:

    * `minItems` to prevent empty arrays
    * `maxItems` to limit size
    * `uniqueItems` to avoid duplicates
* Prefer **arrays of objects** over deeply nested arrays
* Always reuse schemas via `components` where possible
