## 47. Deep Dive into `boolean` Data Type

In this lecture, the focus is on the **`boolean` data type** in OpenAPI and how it behaves differently compared to many programming languages.

---

## What `boolean` Means in OpenAPI

The `boolean` data type is **strict** in OpenAPI.

Only **two values are valid**:

* `true`
* `false`

Anything else is **invalid**, including:

* `"true"` or `"false"` (strings ❌)
* `0` or `1` (numbers ❌)
* `null` (not a boolean ❌)
* empty values ❌

This is different from many programming languages where:

* `0` → false
* `1` → true
* `null` → false

⚠️ **None of these implicit conversions exist in OpenAPI**.

---

## Basic Boolean Property

### YAML Example — Simple Boolean Field

```yaml
schema:
  type: object
  properties:
    isActive:
      type: boolean
```

Valid values:

```json
true
false
```

Invalid values:

```json
"true"
0
null
```

---

## Boolean in a Real Schema (Office Address Example)

From the lecture scenario, an address may or may not be an **office address**.

### YAML Example — Boolean Field in Address Schema

```yaml
Address:
  type: object
  properties:
    street:
      type: string
    city:
      type: string
    isOfficeAddress:
      type: boolean
```

Use case:

* `true` → deliver only during office hours
* `false` → residential delivery rules

Swagger UI will automatically render this as a **boolean toggle**.

---

## Default Boolean Behavior (Important Note)

Swagger UI often **assumes `true` as the default example value** for boolean fields if no example is provided.

This can look confusing, especially if your business logic expects `false` by default.

To avoid confusion, **always provide an explicit example**.

### YAML Example — Boolean with Explicit Example

```yaml
isOfficeAddress:
  type: boolean
  example: false
```

---

## Allowing `null` Values (Nullable Fields)

### Important Concept

OpenAPI **does NOT have a `null` data type**.

Instead, you use:

```yaml
nullable: true
```

This means:

* the field can be `true`
* the field can be `false`
* the field can also be `null`

---

## Boolean with `nullable: true`

### YAML Example — Nullable Boolean

```yaml
isOfficeAddress:
  type: boolean
  nullable: true
```

Swagger UI representation:

* Shows that `null` is a valid value
* Consumers clearly understand optional behavior

---

## Nullable Non-Boolean Fields (From Lecture Example)

The lecture also demonstrated `nullable` with a string field.

### YAML Example — Nullable String

```yaml
categoryName:
  type: string
  nullable: true
```

This indicates:

* `"Electronics"` ✅
* `null` ✅
* empty or missing value ❌ (unless optional)

## Boolean vs File Data Type (Clarification)

Earlier versions of Swagger (OpenAPI 2.x) had a **`file` data type**.

In **OpenAPI 3.x**:

* `file` data type is **removed**
* Files are represented using `string` + `format`

This is **not related to boolean**, but the lecture clarifies version differences.

### YAML Example — File Upload (Reminder)

```yaml
file:
  type: string
  format: binary
```

---

## Common Mistakes to Avoid

❌ Using strings instead of booleans

```yaml
type: "true"   # wrong
```

❌ Assuming numeric booleans

```yaml
example: 1     # wrong
```

❌ Forgetting `nullable` when `null` is expected

```yaml
type: boolean  # null not allowed unless nullable is true
```

---

## Best Practices for Boolean Fields

* Always use **pure `true` / `false`**
* Provide an **explicit example**
* Use `nullable: true` when `null` is valid
* Do not rely on language-specific truthy/falsey rules
* Make boolean field names **self-explanatory**

    * `isActive`
    * `hasDiscount`
    * `isOfficeAddress`

---

## Key Takeaway

> **OpenAPI booleans are strict, explicit, and predictable.
> No magic values, no implicit conversions — only `true` and `false`.**

This clarity is exactly what makes OpenAPI contracts reliable for tools, generators, and consumers.
