## 48. Deep Dive into **Object** Data Type

In this lecture, we focus on the **`object` data type**, which is the backbone of most real-world OpenAPI schemas.
Almost everything meaningful in an API—**products, orders, addresses, users**—is modeled using objects.

---

## Defining an Object with Properties

When you declare `type: object`, you typically describe its structure using the **`properties`** keyword.

### YAML Example — Basic Object Schema

```yaml
Product:
  type: object
  properties:
    id:
      type: integer
    name:
      type: string
    price:
      type: number
```

Key points:

* `properties` defines the **fields** inside the object
* Each property must define its own `type`
* By default, **all properties are optional**

---

## Making Properties Mandatory (`required`)

OpenAPI treats **all object properties as optional by default**.
To make fields mandatory, you must explicitly list them under `required`.

### YAML Example — Required Properties

```yaml
Product:
  type: object
  required:
    - id
    - name
  properties:
    id:
      type: integer
    name:
      type: string
    price:
      type: number
```

Important rules:

* `required` is an **array**
* You must list **property names**, not property definitions
* An empty `required: []` is **invalid**
* `required` must be at the **same level** as `properties`

---

## Read-Only Properties (`readOnly: true`)

A **read-only** property:

* Appears **only in responses**
* Must **not** be sent by consumers in requests

Typical use cases:

* IDs generated by backend
* Timestamps
* System-managed fields

### YAML Example — Read-Only Field

```yaml
Product:
  type: object
  properties:
    id:
      type: integer
      readOnly: true
    name:
      type: string
    releaseDate:
      type: string
      format: date
      readOnly: true
```

Effect:

* `id` and `releaseDate` appear in **GET responses**
* They are **excluded** from POST / PUT request bodies in Swagger UI

---

## Write-Only Properties (`writeOnly: true`)

A **write-only** property:

* Appears **only in requests**
* Never returned in responses

Typical use cases:

* Passwords
* Security tokens
* Sensitive input fields

### YAML Example — Write-Only Field

```yaml
User:
  type: object
  properties:
    username:
      type: string
    password:
      type: string
      format: password
      writeOnly: true
```

Effect:

* `password` is accepted in request bodies
* `password` is **never shown** in responses

---

## Read-Only vs Write-Only (Quick Comparison)

| Attribute   | Request | Response |
|-------------|---------|----------|
| `readOnly`  | ❌       | ✅        |
| `writeOnly` | ✅       | ❌        |

They are **mutually exclusive** in intent and behavior.

---

## Free-Form Objects (Dynamic Objects)

Sometimes, you want an object to accept **any structure**, without predefined properties.

This is called a **free-form object**.

### Option 1 — Object Without Properties

```yaml
metadata:
  type: object
```

### Option 2 — Using `additionalProperties: true`

```yaml
metadata:
  type: object
  additionalProperties: true
```

Both mean:

* Any keys allowed
* Any value types allowed

⚠️ **Not recommended for most APIs**, as it weakens contracts and validation.

---

## Restricting Free-Form Objects

If you must allow free-form objects, you can still apply limits.

### YAML Example — Property Count Restrictions

```yaml
metadata:
  type: object
  additionalProperties: true
  minProperties: 1
  maxProperties: 10
```

This limits:

* Minimum number of fields
* Maximum number of fields

---

## Best Practice: Objects Inside `components/schemas`

Instead of defining object schemas inline everywhere, you should define them once under **`components.schemas`** and
reuse them.

### YAML Example — Object in Components

```yaml
components:
  schemas:
    Address:
      type: object
      required:
        - street
        - city
      properties:
        street:
          type: string
        city:
          type: string
        postalCode:
          type: string
```

### Referencing the Object

```yaml
schema:
  $ref: '#/components/schemas/Address'
```

Benefits:

* No duplication
* Single place to update
* Cleaner OpenAPI documents
* Easier maintenance

---

## Summary of Object Keywords Covered

| Keyword                | Purpose                |
|------------------------|------------------------|
| `type: object`         | Declares an object     |
| `properties`           | Defines object fields  |
| `required`             | Marks mandatory fields |
| `readOnly`             | Response-only field    |
| `writeOnly`            | Request-only field     |
| `additionalProperties` | Allows dynamic fields  |
| `minProperties`        | Minimum field count    |
| `maxProperties`        | Maximum field count    |

---

## Key Takeaway

> **Objects define structure, contracts, and clarity in OpenAPI.
> Use `required`, `readOnly`, and `writeOnly` wisely to express real business rules—not just data shapes.**
