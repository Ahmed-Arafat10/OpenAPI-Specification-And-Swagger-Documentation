## 50. Deep Dive into `enum` Keyword

The `enum` keyword is **not a data type** by itself.
Instead, it is a **constraint** that can be applied to **any data type** to explicitly define a **fixed list of allowed values**.

Using `enum` improves:

* API documentation clarity
* Consumer experience
* Validation (both human and tool-driven)
* Generated client/server code correctness

---

## Why `enum` Is Important

Without `enum`, consumers may:

* Guess allowed values
* Send invalid inputs
* Rely on assumptions instead of documentation

With `enum`, you explicitly say:

> “These are the **only values** I accept. Anything else is invalid.”

---

## Enum Can Be Used With Any Data Type

You can apply `enum` to:

* `string`
* `integer`
* `number`
* `boolean` (rare but allowed)
* Even inside request parameters, request bodies, and responses

---

## Example 1: Enum in Query Parameter (Sorting Example)

### Use Case

You want to accept only:

* `ASC` → ascending
* `DESC` → descending

### YAML Example — Query Parameter with Enum

```yaml
parameters:
  - name: sort
    in: query
    required: false
    schema:
      type: string
      enum:
        - ASC
        - DESC
    description: Sort order of the results
```

**Effect**

* Consumers clearly see valid values
* Swagger UI shows a dropdown
* Any value outside `ASC` or `DESC` is invalid

---

## Example 2: Enum Inside Schema Property (State Restriction)

### Use Case

Easy Shop delivers orders **only in selected states**.

### YAML Example — Enum in Object Schema

```yaml
components:
  schemas:
    Address:
      type: object
      properties:
        state:
          type: string
          enum:
            - California
            - Texas
            - Indiana
            - New York
```

**Meaning**

* Orders are accepted only from these states
* Consumers cannot guess unsupported states
* Validation rules become explicit

---

## Example 3: Enum with Integer Values

Enums are **not limited to strings**.

### YAML Example — Integer Enum

```yaml
type: integer
enum:
  - 101
  - 102
  - 103
```

Useful for:

* Category IDs
* Status codes
* Fixed numeric identifiers

---

## Example 4: Enum with `null` as a Valid Value

If `null` is allowed, you must **explicitly include it**.

### YAML Example — Allowing `null`

```yaml
type: string
enum:
  - ACTIVE
  - INACTIVE
  - null
```

Without explicitly listing `null`, it is **not allowed**.

---

## Enum + Nullable (Alternative)

Another way to allow `null` is using `nullable: true`.

```yaml
type: string
enum:
  - ACTIVE
  - INACTIVE
nullable: true
```

Both approaches are valid, but **explicit `null` in enum is clearer** for consumers.

---

## How Enum Appears in Swagger UI

* Rendered as a **dropdown**
* Prevents free-text input
* Makes APIs easier to explore
* Reduces documentation questions

---

## Best Practices for Using `enum`

✔ Use `enum` whenever values are **finite and known**
✔ Prefer `enum` over textual explanations in `description`
✔ Keep enum values **meaningful and consistent**
✔ Avoid enums with too many values (consider lookup APIs instead)
✔ Always ensure enum values **match the declared type**

---

## Common Mistakes to Avoid

❌ Using values not matching the type
❌ Forgetting to include `null` when needed
❌ Using enums where values are dynamic
❌ Mixing case inconsistently (`asc` vs `ASC`)

---

## Key Takeaways

* `enum` restricts values, it is **not a data type**
* Works with **any schema type**
* Improves validation, documentation, and UI
* Must explicitly include `null` if allowed
* Strongly recommended for query params, filters, statuses, and categories

This completes the deep dive into the **`enum` keyword** in OpenAPI specifications.
