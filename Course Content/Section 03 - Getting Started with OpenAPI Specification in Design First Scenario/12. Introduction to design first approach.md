# 12. Introduction to Design-First Approach

## 1. Purpose of This Lecture

This lecture introduces the **Design-First approach** for OpenAPI Specification, which is the **opposite workflow** of the Code-First approach discussed earlier.

Here, the focus is on:

* Designing APIs **before** implementation
* Writing OpenAPI Specification **manually**
* Using the specification as the **foundation** for development, collaboration, and automation

---

## 2. When Design-First Approach Is Used

In many real-world scenarios:

* Clients provide **business-level requirements**, not technical APIs
* Example:

    * “This is my business scenario”
    * “I want APIs for X, Y, Z”
    * “These APIs should be exposed to these consumers”

At this stage:

* No code exists
* Only ideas, workflows, and requirements exist

This is where the **Design-First approach** fits naturally.

---

## 3. Collaborative API Design Phase

Based on business requirements:

* Multiple stakeholders collaborate:

    * Project managers
    * Business analysts
    * Developers
    * Testers

During these discussions:

* The team visualizes how the system should behave
* API contracts are shaped
* **OpenAPI Specification is written alongside design discussions**

A key point emphasized:

> Someone in the team **must take ownership** of writing the OpenAPI Specification during design.

---

## 4. Why Writing OpenAPI Early Is Powerful

When OpenAPI Specification is written during the design phase:

* Backend developers can:

    * Generate server-side skeleton code
    * Focus only on business logic

* Frontend / consumer teams can:

    * Start development immediately
    * Avoid waiting for backend implementation

* QA teams can:

    * Understand expected behavior early
    * Prepare test cases in advance

This enables **parallel development** across teams.

---

## 5. Design-First Is Highly Recommended for Product Companies

For product companies exposing APIs to:

* JavaScript
* Java
* Python
* .NET
* Other ecosystems

Design-First OpenAPI provides massive value because:

* One OpenAPI spec → many languages
* Client SDKs and server stubs can be generated automatically
* Writing and maintaining SDKs manually becomes unnecessary

Without OpenAPI:

* Supporting multiple languages becomes a **huge organizational burden**

---

## 6. Responsibility of the OpenAPI Specification

In Design-First:

* OpenAPI Specification must be:

    * Carefully written
    * Complete
    * Accurate

Once the team feels:

* All scenarios are covered
* All request/response formats are correct

Then:

* The same specification can be used by:

    * API producers
    * API consumers

This makes the specification the **single source of truth**.

---

## 7. Manual Writing Challenge

A key challenge with Design-First:

* OpenAPI must be written **letter by letter**
* Developers may not know:

    * The correct structure
    * Required elements
    * Mandatory fields
* Syntax mistakes are easy to make

This makes manual writing:

* Error-prone
* Intimidating at first

---

## 8. Tool Support: Swagger Editor

To solve these problems, the course introduces **Swagger Editor**.

Swagger Editor:

* Runs directly in the browser
* Supports YAML and JSON
* Provides **inline validation**
* Immediately highlights:

    * Syntax errors
    * Structural issues
    * Missing or invalid fields

This makes Design-First OpenAPI writing:

* Faster
* Safer
* More beginner-friendly

---

## 9. YAML vs JSON for OpenAPI

OpenAPI Specification supports:

* YAML
* JSON

For this course:

* **YAML is chosen**
* Reasons:

    * More readable
    * Less verbose
    * More commonly used in OpenAPI projects

All upcoming OpenAPI examples will be written in **YAML**.
