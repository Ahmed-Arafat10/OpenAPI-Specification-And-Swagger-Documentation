# 13. Quick Introduction to the History of YAML

## 1. Purpose of This Lecture

This lecture gives a **quick historical and practical background of YAML** so you feel comfortable reading and writing YAML while defining **OpenAPI Specifications** in upcoming lectures.

The goal is **not mastery**, but:

* Understanding *why* YAML exists
* Knowing *why it is preferred* over XML and JSON for OpenAPI
* Becoming comfortable with its **basic philosophy and syntax style**

---

## 2. Data Representation Formats Used in the Industry

Historically, the industry has mainly used **three formats** for data exchange and configuration:

* **XML**
* **JSON**
* **YAML**

Each emerged to solve problems of the previous one.

![Image](https://www.devopsschool.com/blog/wp-content/uploads/2021/07/Comparison-XML-JSON-YAML-3.png)

---

## 3. XML: The Early Standard (Around 2000)

### 3.1 Why XML Was Popular

* Widely used for:

    * Data communication
    * Server configurations
    * Enterprise integrations
* Strongly structured
* Self-descriptive via tags

### 3.2 Major Problems with XML

XML requires:

* Opening tags
* Closing tags
* Repeated metadata

Example issues:

* More metadata than actual data
* Extremely verbose
* Heavy payloads over the network

In large systems:

* XML files can grow to thousands of lines
* Data transfer becomes inefficient

This made XML **unsuitable for lightweight REST APIs and mobile applications**.

---

## 4. JSON: Lightweight Alternative for APIs

### 4.1 Why JSON Replaced XML for APIs

JSON (JavaScript Object Notation) solved many XML problems:

* No opening/closing tags
* Less metadata
* Smaller payload size
* Easy representation of:

    * Objects
    * Arrays
    * Nested structures

This made JSON ideal for:

* REST APIs
* Mobile-to-server communication

---

### 4.2 Limitations of JSON for Humans

Although JSON is machine-friendly, it is **human-error-prone**:

* Strict syntax rules:

    * Double quotes
    * Commas
    * Colons
    * Brackets
* Missing a single character makes JSON invalid

This became painful when:

* Developers wrote large configuration files
* DevOps and cloud configurations increased rapidly

---

## 5. YAML: Human-Friendly Configuration Format

### 5.1 Why YAML Came Into Picture

As DevOps and cloud adoption increased:

* Humans started writing configurations more than machines
* JSON became cumbersome for large configs

YAML was introduced to be:

* **Easy to write**
* **Easy to read**
* **Human-friendly**

---

### 5.2 What Makes YAML Different

YAML focuses on:

* **Indentation**
* **Structure**
* **Whitespace**

Instead of:

* Curly braces `{ }`
* Square brackets `[ ]`
* Quotes `" "`
* Commas `,`

YAML uses:

* Spaces
* Indentation levels
* Hyphens (`-`) for lists

This makes YAML:

* Cleaner
* More readable
* Less error-prone for humans

---

## 6. Representing Arrays and Objects in YAML

Key differences from JSON:

* **Arrays**

    * YAML uses `-`
    * JSON uses `[ ]`

* **Objects**

    * YAML uses indentation
    * JSON uses `{ }`

This simplicity is one of the main reasons **OpenAPI prefers YAML**.

---

## 7. Meaning of YAML

**YAML** stands for:

> **YAML Ain’t Markup Language**

This highlights:

* YAML is **not tag-based** like XML
* YAML is focused on **data**, not markup

---

## 8. Why OpenAPI Commonly Uses YAML

OpenAPI supports:

* YAML
* JSON

But YAML is preferred because:

* OpenAPI documents are large
* Readability matters
* Humans maintain these files
* YAML reduces visual and syntactical noise

That’s why **this course uses YAML exclusively**.
