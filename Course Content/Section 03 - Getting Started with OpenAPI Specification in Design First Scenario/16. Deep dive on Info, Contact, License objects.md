# 16. Deep Dive on `info`, `contact`, and `license` Objects

## 1. Context from Previous Lecture

In the previous lecture, we successfully wrote a **minimal OpenAPI Specification document** using only the mandatory root elements:

* `openapi`
* `info`
* `paths`

That document was **valid**, but it contained only the bare minimum.
This lecture expands on that foundation by exploring the **`info` object in depth**, including its **optional but highly important sub-fields**.

This is where your OpenAPI document starts becoming **useful and expressive**, not just valid.

---

## 2. Role of the `info` Object

The `info` object is a **mandatory root element** in any OpenAPI Specification.

Its purpose is to provide **metadata about your API**, such as:

* What the API is about
* Who maintains it
* How consumers can contact you
* Under what license the API can be used
* Which version of the API this document represents

Consumers rely heavily on this section to understand **context**, not just endpoints.

---

## 3. Mandatory Fields Inside `info`

Even though `info` supports many fields, **only two are mandatory**:

### 3.1 `title`

* A short, human-readable name for your API
* Similar to a web page title shown in a browser tab
* Helps consumers immediately understand what the API represents

This field **must always be present**.

---

### 3.2 `version`

* Represents the **version of your API**
* Not related to the OpenAPI version
* Used for API versioning (e.g. `0.0.1`, `1.0.0`, `2.1.3`)

This field is mandatory because:

* Organizations often support multiple API versions
* Consumers must clearly know which version they are using
* Avoids confusion and accidental usage of wrong API versions

---

## 4. Optional but Important Fields in `info`

Although optional, these fields significantly improve API clarity and professionalism.

### 4.1 `summary`

* A **short, concise explanation** of what the API does
* Intended for quick scanning
* Useful in documentation portals and API catalogs

---

### 4.2 `description`

* A **detailed explanation** of the API
* Can include:

    * Business context
    * Functional scope
    * Usage expectations
* Often written in multiple sentences or paragraphs

---

### 4.3 `termsOfService`

* A URL pointing to the **terms and conditions**
* Helps consumers understand:

    * Legal usage rules
    * Restrictions
    * Responsibilities

The value **must be a valid URL**.

---

## 5. The `contact` Object

The `contact` field is an **object**, not a simple value.

Its purpose is to tell API consumers:

> “If you have questions or issues, this is how you reach us.”

### 5.1 Fields Inside `contact`

All fields inside `contact` are optional, but highly recommended:

* **`name`**
  Name of the team or individual responsible for the API

* **`url`**
  A support page, issue tracker, or help desk URL

* **`email`**
  Email address where consumers can send questions or issues

This object improves:

* Developer experience
* Support workflows
* Trust in your API

---

## 6. The `license` Object

The `license` field is also an **object**.

It describes **how your API can be legally used**.

### 6.1 When `license` Is Present

* The `license` object itself is optional
* But **if you include it**, then:

    * `name` becomes **mandatory**

This is a critical rule.

---

### 6.2 Fields Inside `license`

* **`name`** (mandatory if `license` exists)
  Name of the license (e.g. MIT, Apache 2.0, Proprietary)

* **`identifier`**
  A standard license identifier (if applicable)

* **`url`**
  A URL pointing to full license terms

This ensures consumers clearly understand:

* Whether the API is free
* Commercial restrictions
* Redistribution rules

---

## 7. Why These Fields Matter in Real Projects

Adding rich metadata inside `info`:

* Makes APIs easier to adopt
* Reduces back-and-forth communication
* Improves documentation portals
* Helps large organizations manage multiple APIs cleanly

In enterprise and public APIs, **`contact` and `license` are strongly expected**, even though they are technically optional.
