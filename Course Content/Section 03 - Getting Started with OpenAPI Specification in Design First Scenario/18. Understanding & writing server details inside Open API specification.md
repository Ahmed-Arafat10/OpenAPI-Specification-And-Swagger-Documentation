# 18. Understanding & Writing `servers` Details Inside OpenAPI Specification

## 1. Purpose of This Lecture

This lecture introduces the **`servers` root element** in OpenAPI Specification and explains how API consumers can discover **where** and **how** to access your APIs.

Until now:

* We defined **metadata** (`info`)
* But we did not tell consumers **which URLs** to use

The `servers` object solves that problem.

---

## 2. What Is the `servers` Root Element?

`servers` is a **root-level element** in OpenAPI Specification used to define:

* API base URLs
* Environment-specific endpoints (dev, staging, prod, etc.)
* Optional URL templating via variables

Key points:

* `servers` is **optional**
* Not mandatory by design

---

## 3. Why `servers` Is Optional

Some organizations choose **not** to expose server details because:

* Hostnames may be sensitive
* Infrastructure may change frequently
* Security policies forbid publishing endpoints

In such cases:

* Server details may be shared privately
* Or overridden dynamically by consumers

But **whenever possible**, defining `servers` is highly recommended for usability.

---

## 4. Structure of the `servers` Object

`servers` is an **array of objects**.

In YAML:

* Arrays are represented using `-` (hyphen)

Each server entry can contain:

* `url` (mandatory)
* `description` (optional but recommended)
* `variables` (optional, advanced)

---

## 5. Basic `servers` Example (Multiple Environments)

A common real-world setup includes:

* Development
* Staging
* Production

Example added in this lecture:

```yaml
servers:
  - url: https://dev.easyshop.com
    description: Development server

  - url: https://staging.easyshop.com
    description: Staging server

  - url: https://api.easyshop.com
    description: Production server
```

Effect in Swagger UI:

* Consumers see a dropdown of available servers
* They can switch environments easily

---

## 6. YAML Indentation Rules for `servers`

Important clarification from the lecture:

* Indentation under `servers` is **recommended for readability**
* YAML can still parse lists without indentation
* But consistent indentation avoids confusion and mistakes

Best practice:

* Always indent list items under `servers`

---

## 7. Advanced Usage: Server Variables

OpenAPI allows **URL templating** using variables.

Instead of hardcoding values in the URL, you can use placeholders.

Example placeholders:

* `{username}`
* `{port}`
* `{version}`

These placeholders must be defined under `variables`.

---

## 8. Server Variables Example

Example introduced in the lecture:

```yaml
servers:
  - url: https://{username}:{port}/{version}
    description: Custom server with variables
    variables:
      username:
        default: demo
        description: Username for authentication

      port:
        default: 8443
        enum:
          - 8443
          - 443

      version:
        default: v1
```

---

## 9. Understanding Each Variable Field

### 9.1 `default`

* Mandatory for variables
* Used to compute the default server URL
* Displayed automatically in Swagger UI

### 9.2 `description`

* Helps consumers understand the purpose of the variable
* Optional but recommended

### 9.3 `enum`

* Restricts allowed values
* Swagger UI enforces selection
* Prevents invalid values

Example behavior:

* Port can only be `8443` or `443`
* User cannot type arbitrary values

---

## 10. How Swagger UI Uses Server Variables

Swagger UI:

* Displays placeholders initially
* Computes a final URL using default values
* Provides input fields for variables
* Dynamically updates the computed server URL

Behavior highlights:

* Variables with `enum` â†’ dropdown
* Variables without `enum` â†’ text input
* Changing values updates the computed URL instantly

When switching to a server **without variables**:

* Variable inputs disappear automatically

## 11. Order and Placement Rules

* `servers` is a **root element**
* Typically placed:

    * After `info`
    * Before `paths`

However:

* Order of root elements **does not affect validity**
* Placement is for readability and convention only
