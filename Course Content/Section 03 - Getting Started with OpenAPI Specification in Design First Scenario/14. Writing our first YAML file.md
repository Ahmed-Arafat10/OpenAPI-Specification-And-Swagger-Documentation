## 3. Core YAML Principle: Key–Value Structure

Everything in YAML is written as **key : value**.

Example concept:

* `name` → key
* `easy bytes` → value

Important characteristics:

* No curly braces `{ }`
* No mandatory double quotes
* Much cleaner than JSON

YAML is smart enough to:

* Interpret strings without quotes
* Accept quotes if you choose to include them

---

## 4. Writing Nested Objects Using Indentation

YAML represents nested objects using **indentation**, not brackets.

### 4.1 Parent Object

Example concept:

* `person` acts as a parent key

To define values inside `person`:

* Press **Enter**
* Add **indentation (spaces)**

### 4.2 Child Fields

Inside `person`, you can define:

* Name
* Age
* Other attributes

Indentation rule:

* All child keys must be aligned at the same indentation level

This works similarly to **block indentation in Python**.

---

## 5. Writing Deeper Nested Objects

You can nest objects multiple levels deep.

Example concept:

* `person`

    * `address`

        * `addressLine1`
        * `city`
        * `country`

Each deeper level:

* Requires additional indentation
* Belongs strictly to its parent key

If indentation is wrong:

* The structure changes
* The data moves to a different parent

---

## 6. Writing Lists (Arrays) in YAML

YAML uses **hyphens (`-`)** to represent lists.

### 6.1 List Structure

To define a list:

* Create a key (e.g., `courses`)
* Indent
* Start each item with `-`

This replaces:

* Square brackets `[ ]`
* Comma-separated values

### 6.2 Importance of Indentation in Lists

Indentation controls **ownership**:

* If a list is indented under `person`, it belongs to `person`
* If aligned with top-level keys, it belongs to the root

A single extra space can:

* Completely change the data hierarchy

---

## 7. Space After Colon Rule (Critical)

When writing a key and value on the **same line**, YAML requires:

* A **space after the colon**

Correct:

```
name: John
```

Incorrect:

```
name:John
```

Without the space:

* YAML cannot identify the key properly
* The file becomes invalid

This is one of the **most common YAML mistakes**.

---

## 8. Why YAML Is Easier Than JSON

Compared to JSON:

* No brackets
* No commas
* No closing syntax
* Less visual noise
* Lower chance of syntax errors

JSON requires:

* Strict punctuation
* Perfect structure

YAML requires:

* Correct indentation
* Logical structure

This makes YAML:

* Faster to write
* Easier to read
* More forgiving for humans

---

## 9. YAML for Configurations & OpenAPI

Because YAML is:

* Less error-prone
* More readable
* Easier to maintain

It is heavily used in:

* DevOps configurations
* Cloud platforms (AWS, GCP, Azure)
* CI/CD pipelines
* OpenAPI Specification files

That is why **this course uses YAML** instead of JSON for OpenAPI.

---

## 10. YAML as a Superset of JSON

YAML is a **superset of JSON**, which means:

* Everything valid in JSON can be represented in YAML
* YAML can always be converted to JSON

If YAML contains features not supported by JSON:

* They are ignored during conversion

This makes YAML a **safe choice** even when JSON output is required.

---

## 11. Comments in YAML

YAML supports **comments**, JSON does not.

* Comments start with `#`
* Used for explanations or documentation
* Ignored during conversion to JSON

When converting YAML → JSON:

* Comments are removed automatically

This makes YAML excellent for:

* Human-readable documentation
* Configuration explanation
* API design clarity

---

## 12. Practical Recommendation from the Lecture

* Prefer **YAML** over JSON for OpenAPI
* Use online converters while learning
* Focus on:

    * Indentation
    * Structure
    * Readability

Advanced YAML features are **not required immediately** and will be introduced gradually when needed.

```yaml
# sample yaml file

name: easy bytes

person:
  name: John Doe
  age: 30
  address:
    addressLine1: street number one
    city: New York
    country: USA

courses:
  - YAML
  - OpenAPI
  - Spring
  - Microservices
```

### Notes (just for clarity, not part of YAML logic)

* Indentation defines hierarchy
* `-` represents list items
* Comments start with `#`
* No quotes, commas, or brackets are required
* This YAML can be safely converted to JSON
