# 17. Writing `info`, `contact`, and `license` Details Inside OpenAPI Specification

## 1. Goal of This Lecture

In this lecture, the previously created **minimal OpenAPI Specification** is enhanced by **adding richer metadata** under the `info` object.
All elements discussed theoretically in the last lecture are now **implemented practically** using **YAML** inside Swagger Editor.

The focus is on:

* Updating the `info` object
* Handling version compatibility issues
* Correct YAML indentation and case sensitivity
* Seeing how changes immediately reflect in Swagger UI

## 2. Important Note About Swagger Editor (Browser Version)

The browser-based **Swagger Editor** has a useful behavior:

* It **persists your last edited OpenAPI document**
* Even after:

    * Refreshing the page
    * Revisiting after days or weeks
* As long as browser cache is not cleared, the document remains

### Saving Your Work Manually

If you want local backups:

* Use **File → Save as YAML**
* Or convert and download as **JSON**

This is useful for:

* Versioning
* GitHub commits
* Offline storage

## 3. Adding More Fields to the `info` Object

Previously, only mandatory fields were present:

* `title`
* `version`

This lecture adds **optional but valuable fields**.

---

## 4. Version Compatibility Issue (`summary` Field)

When attempting to add `summary`:

* Swagger Editor throws an error
* It marks `summary` as an **additional / unsupported property**

### Reason

* `summary` is supported in **OpenAPI 3.1.x**
* Swagger Editor (at the time of recording) supports **up to 3.0.3**

Because the document uses:

```yaml
openapi: 3.0.3
```

The `summary` field **must be removed** for compatibility.

> When Swagger Editor fully supports 3.1.x, `summary` can be safely added.

---

## 5. Using `description` (Supported Alternative)

Instead of `summary`, `description` is used.

* `description` is fully supported in OpenAPI 3.0.3
* Accepts:

    * Single-line text
    * Multi-line text

### Multi-line Description Using YAML Pipe (`|`)

To write long descriptions:

* Use the pipe symbol `|`
* Write content across multiple lines

YAML treats all indented lines under `|` as the value of the key.

This is ideal for:

* Detailed API explanations
* Business context
* Usage notes

---

## 6. Adding `termsOfService`

* `termsOfService` expects a **valid URL**
* Appears in Swagger UI as a clickable link
* Allows consumers to:

    * Read legal terms
    * Understand usage rules

### Case Sensitivity Warning

* YAML + OpenAPI are **case-sensitive**
* `termsOfService` ❌
* `TermsOfService` ❌
* `termsofservice` ❌

Only the correct casing works.

---

## 7. Defining the `contact` Object

`contact` is an **object**, not a single value.

### Structure Rules

* Declare `contact:`
* Press Enter
* Indent child fields

Fields added:

* `name`
* `url`
* `email`

### Indentation Is Critical

If indentation is incorrect:

* Fields may accidentally become part of `info`
* The specification becomes invalid

Correct indentation ensures:

* `contact` belongs to `info`
* Child fields belong to `contact`

---

## 8. Defining the `license` Object

`license` is also an **object**.

Fields used:

* `name`
* `url`

### Naming Best Practice

Choose a meaningful `name`:

* Avoid vague values like `EasyShop 2.0`
* Prefer descriptive names like:

    * `Easy Shop License`

This ensures Swagger UI users clearly understand:

* The link relates to licensing
* Not just a random resource

---

## 9. Order of Fields Does NOT Matter

OpenAPI does **not enforce ordering**:

* Fields inside `info` can be rearranged
* Fields inside `contact` or `license` can be reordered

Valid examples:

* `contact` before `license`
* `license` before `contact`
* `email` before `name`

What **does matter**:

* Correct field names
* Correct casing
* Correct indentation
* Valid data types

---

## 10. Final OpenAPI `info` Section (As Written in This Lecture)

```yaml
openapi: 3.0.3

info:
  title: Easy Shop Products API Definition
  description: |
    Using this OpenAPI specification,
    any consumer can understand the APIs
    that are exposed by Easy Shop.
  termsOfService: https://easyshop.com/terms
  contact:
    name: Easy Shop Support
    url: https://easyshop.com/support
    email: support@easyshop.com
  license:
    name: Easy Shop License
    url: https://easyshop.com/license
  version: 0.0.21

paths: {}
```
