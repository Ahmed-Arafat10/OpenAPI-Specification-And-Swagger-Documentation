# 15. Writing a Minimal OpenAPI Specification Document

## 1. Purpose of This Lecture

This lecture introduces the **smallest possible OpenAPI Specification document** that is still **valid**.
The goal is to:

* Understand the **mandatory structure**
* Learn which elements are **required vs optional**
* Write your **first valid OpenAPI document in YAML**
* Get comfortable using **Swagger Editor** for validation and visualization

This is the **first real milestone** in your OpenAPI journey.

---

## 2. Why Start with a Minimal Specification?

OpenAPI Specification is:

* Large
* Structured
* Detailed

Trying to learn everything at once is overwhelming.
So the course intentionally:

* Breaks the specification into **small pieces**
* Explains **one element at a time**
* Builds the document incrementally

This lecture focuses **only** on what is *absolutely required*.

---

## 3. Mandatory Elements of a Valid OpenAPI Document

To be considered **valid**, an OpenAPI document must contain **three root-level elements**:

### 3.1 `openapi`

* **Mandatory**
* Must be the **first root element**
* Defines the **OpenAPI version**, not your API version

Example values:

* `3.0.3`
* `3.1.x`

> This tells tools how to interpret the document.

---

### 3.2 `info`

* **Mandatory**
* An object that describes your API at a high level
* Used by humans and tools

Inside `info`, **two fields are mandatory**:

* `title` → short description of your API
* `version` → version of *your API*, not OpenAPI

---

### 3.3 `paths`

* **Mandatory**
* Describes all API endpoints
* Can be **empty** in a minimal document

An empty `paths` object means:

> “This API has no defined operations yet.”

---

## 4. Important Version Clarification (Swagger vs OpenAPI)

The Swagger Editor initially shows an example using:

```
swagger: "2.0"
```

This is **old**.

Key rule:

* **Swagger 2.x** → uses `swagger`
* **OpenAPI 3.x+** → uses `openapi`

From OpenAPI 3.0 onward:

* You **must not** use `swagger`
* You **must** use `openapi`

This change happened after Swagger was donated to the open-source community.

---

## 5. Choosing the OpenAPI Version

Although newer OpenAPI versions exist (e.g. 3.1.x):

* Swagger tools may **lag behind**
* Not all tools immediately support the latest version

So for maximum compatibility in this course:

* **OpenAPI 3.0.3** is used

> If your tools support 3.1.x later, you can safely upgrade.

## 7. Writing the Minimal OpenAPI Document (YAML)

Below is the **exact minimal OpenAPI Specification** written in this lecture:

```yaml
openapi: 3.0.3

info:
  title: Easy Shop Products API Definition
  version: 0.0.21

paths: {}
```

### What This Means

* The document is **valid**
* No API operations are defined yet
* Swagger UI correctly renders it
* No validation errors remain

Swagger UI will display:

> “No operations defined in spec!”

This is **expected and correct**.

---

## 8. Understanding the Two Versions Used

Important distinction:

* `openapi: 3.0.3`
  → Version of the **OpenAPI Specification**

* `info.version: 0.0.21`
  → Version of **your API**

They serve **different purposes** and must not be confused.

---

## 9. Why `paths` Can Be Empty (For Now)

Even though `paths` is mandatory:

* It does **not** need to contain endpoints yet
* An empty object `{}` is valid

This allows you to:

* Start with structure
* Add endpoints later
* Keep validating incrementally

---

## 10. Editor + UI Working Together

Inside Swagger Editor:

* **Left side** → OpenAPI YAML (Editor)
* **Right side** → Visual API documentation (Swagger UI)

As soon as:

* Required fields are present
* Syntax is correct

Both sides sync instantly.

This feedback loop is critical when learning OpenAPI.
