## 54. Polymorphism using `anyOf` keyword

In this lecture, the goal is to understand **polymorphism using `anyOf`** and how it differs from `oneOf`.

Where `oneOf` enforces **exactly one schema**, `anyOf` allows **one or more schemas** to be valid **at the same time**.

---

## Core Idea of `anyOf`

`anyOf` tells OpenAPI:

> “The data may conform to **any one schema or any combination of the schemas listed**.”

This makes it suitable when:

* The response shape **changes based on input**
* Multiple schemas may appear **together** in a single response

---

## Scenario Used in the Lecture

A new API is introduced:

* API name: **anyOfOrder**
* HTTP Method: `GET`
* Tag: `Orders`
* Input:

    * `orderId` (query parameter, reused from components)
    * `fetchType` (query parameter)

        * Valid values:

            * `summary`
            * `details`

### Business Meaning

* If `fetchType = summary`
  → return **only order summary**
* If `fetchType = details`
  → return **order summary + order address**

This means the **response structure varies**, and sometimes includes **both schemas together**.

That is exactly where `anyOf` fits.

---

## YAML Code – Path Using `anyOf`

```yaml
paths:
  /anyOfOrder:
    get:
      tags:
        - Orders
      summary: Fetch order using anyOf
      description: >
        Returns order summary or full order details based on fetchType.
      parameters:
        - $ref: '#/components/parameters/OrderId'
        - name: fetchType
          in: query
          required: true
          description: >
            summary returns product details only,
            details returns product details with address.
          schema:
            type: string
            enum:
              - summary
              - details
      responses:
        '200':
          description: Order fetched successfully
          content:
            application/json:
              schema:
                anyOf:
                  - $ref: '#/components/schemas/OrderSummary'
                  - $ref: '#/components/schemas/OrderAddress'
```

---

## Schemas Used by `anyOf`

### OrderSummary Schema

```yaml
components:
  schemas:
    OrderSummary:
      type: object
      properties:
        products:
          type: array
          items:
            $ref: '#/components/schemas/Product'
      required:
        - products
```

### OrderAddress Schema

```yaml
components:
  schemas:
    OrderAddress:
      type: object
      properties:
        address:
          $ref: '#/components/schemas/Address'
      required:
        - address
```

---

## How Validation Works

### Valid Responses

* ✔ Only `OrderSummary`
* ✔ Only `OrderAddress`
* ✔ `OrderSummary` + `OrderAddress` together

### Invalid Responses

* ✘ Response that matches **none** of the schemas

This is the **key difference from `oneOf`**.

---

## `oneOf` vs `anyOf` (Conceptual Difference)

| Keyword | Allowed Schemas           |
|---------|---------------------------|
| `oneOf` | Exactly **one** schema    |
| `anyOf` | **One or more** schemas   |
| `allOf` | **All** schemas combined  |
| `not`   | Schema must **not** match |

---

## Mental Model

* Use **`oneOf`** when:

    * Request/response must match **only one shape**
* Use **`anyOf`** when:

    * Request/response may include **multiple shapes together**
    * Output varies based on input conditions

---

## Important Design Insight (Added by me – not mentioned in the lecture)

In real-world APIs:

* `anyOf` is **most commonly used in responses**
* `oneOf` is **more common in request bodies**

This is because:

* Clients often *send one clear intent*
* Servers often *return variable data based on context*

---

With this lecture, you have now covered **all four advanced schema composition keywords**:

* `allOf` → inheritance
* `oneOf` → strict polymorphism
* `anyOf` → flexible polymorphism
* `not` → exclusion

These are some of the **most powerful and advanced features** in OpenAPI and are heavily used in mature API designs.
