## 51. Introduction to Inheritance & Polymorphism inside OpenAPI Specification

In this lecture, the focus shifts to a **more advanced modeling concept** in OpenAPI:
how to represent **inheritance** and **polymorphism** in API schemas.

These concepts are essential when:

* You want to **reuse common properties** across multiple schemas
* Your API can **accept or return different shapes of data** depending on the scenario

OpenAPI supports these ideas using special schema composition keywords:

* `allOf`
* `oneOf`
* `anyOf`
* `not`

This lecture is **conceptual** and intentionally light. The detailed usage and practical examples are covered in the
upcoming lectures.

---

## Why Inheritance & Polymorphism Are Needed in OpenAPI

As APIs grow, schemas often:

* Share common fields (e.g., `id`, `createdAt`, `status`)
* Vary slightly depending on context (e.g., different request/response structures)

Without inheritance:

* You repeat schema definitions
* Maintenance becomes difficult
* Changes must be applied in multiple places

Without polymorphism:

* You are forced into a single rigid schema
* APIs become less expressive and less flexible

---

## Inheritance (Conceptual Overview)

Inheritance means:

> One schema **inherits** properties from another schema.

### Programming Analogy

In programming languages:

* A **base class** contains common fields
* Other classes **extend** that base class

### OpenAPI Equivalent

* You define a **base schema**
* Other schemas reuse it using **composition keywords**

This avoids duplication and keeps schema definitions consistent.

---

## Polymorphism (Conceptual Overview)

Polymorphism means:

> The same API endpoint can **accept or return multiple schema shapes**.

### Common Scenarios

* API response differs based on success or error
* Request body structure varies by input type
* One endpoint supports multiple resource variants

In OpenAPI, this is achieved by allowing **multiple possible schemas** instead of a single one.

---

## Keywords Introduced in This Lecture

This lecture introduces the purpose of the following keywords (details come later):

### `allOf`

* Used primarily for **inheritance**
* Combines multiple schemas into one
* Child schema includes all parent properties

### `oneOf`

* Used for **polymorphism**
* Exactly **one** schema must match

### `anyOf`

* Used for **polymorphism**
* **At least one** schema must match

### `not`

* Excludes a schema
* Used in advanced validation scenarios

> ⚠️ This lecture explains **why** these keywords exist, not **how** to use them in detail.

---

## High-Level Relationship Between Concepts and Keywords

| Concept      | OpenAPI Keywords |
|--------------|------------------|
| Inheritance  | `allOf`          |
| Polymorphism | `oneOf`, `anyOf` |
| Exclusion    | `not`            |

## YAML Illustration (Conceptual Example)

> This example is for **illustration only**. Full explanations are provided in upcoming lectures.

### Example: Inheritance Using `allOf`

```yaml
components:
  schemas:
    BaseProduct:
      type: object
      properties:
        id:
          type: integer
        name:
          type: string

    DigitalProduct:
      allOf:
        - $ref: '#/components/schemas/BaseProduct'
        - type: object
          properties:
            downloadUrl:
              type: string
```

**What this represents**

* `DigitalProduct` inherits `id` and `name`
* Adds `downloadUrl` on top
* Avoids redefining common fields

---

### Example: Polymorphism Using `oneOf`

```yaml
components:
  schemas:
    PaymentResponse:
      oneOf:
        - $ref: '#/components/schemas/CardPayment'
        - $ref: '#/components/schemas/UPIPayment'
```

**What this represents**

* API response can be **either** `CardPayment` **or** `UPIPayment`
* Exactly one schema must match
