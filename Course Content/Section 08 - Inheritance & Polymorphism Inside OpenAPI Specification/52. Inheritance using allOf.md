## 52. Inheritance using `allOf`

In this lecture, the goal is to **achieve inheritance inside OpenAPI schemas** using the `allOf` keyword.

The problem we are solving is **schema duplication**.

You already have a `Product` schema that contains common fields such as:

* id
* name
* price
* category
* quantity
* release date

Now you want to introduce new schemas like:

* `Mobile`
* `Laptop`

Both of these:

* Share **all fields of Product**
* Add **only one or two fields that are specific** to them

Instead of redefining the same Product fields again and again, we use **`allOf`**.

---

### What `allOf` Does (Conceptually)

`allOf` means:

> “This schema is a **combination of all schemas listed here**”

So when you use `allOf`:

* OpenAPI **merges all referenced schemas**
* The resulting schema behaves like **inheritance**
* The child schema gets **all parent properties + its own**

This is the OpenAPI equivalent of:

* `extends` in Java
* inheritance in OOP

---

## Core Rules of `allOf`

* `allOf` **always accepts an array**
* Each array item is:

    * Either a `$ref` to another schema
    * Or an inline schema definition
* All schemas inside `allOf` are **merged into one**

---

## YAML Example – Base Schema (`Product`)

This is the parent schema that will be reused.

```yaml
components:
  schemas:
    Product:
      type: object
      properties:
        productId:
          type: integer
        name:
          type: string
        price:
          type: number
          format: float
        categoryName:
          type: string
        quantity:
          type: integer
        releaseDate:
          type: string
          format: date
      required:
        - productId
        - name
        - price
```

---

## YAML Example – Inheriting with `allOf` (`Mobile`)

`Mobile`:

* Inherits **everything from Product**
* Adds one unique field: `networkType`

```yaml
components:
  schemas:
    Mobile:
      allOf:
        - $ref: '#/components/schemas/Product'
        - type: object
          properties:
            networkType:
              type: string
              enum:
                - 4G
                - 5G
          required:
            - networkType
```

### Resulting Behavior

`Mobile` now contains:

* All Product fields
* Plus `networkType`
* And `networkType` is mandatory

No duplication. One source of truth.

---

## YAML Example – Inheriting with `allOf` (`Laptop`)

`Laptop`:

* Also inherits Product
* Adds a unique field: `ram`

```yaml
components:
  schemas:
    Laptop:
      allOf:
        - $ref: '#/components/schemas/Product'
        - type: object
          properties:
            ram:
              type: string
              enum:
                - 8GB
                - 16GB
                - 32GB
          required:
            - ram
```

---

## Key Observations

* `Product` is **not copied**
* It is **referenced**
* Any future change to `Product`:

    * Automatically applies to `Mobile` and `Laptop`
* This drastically improves:

    * Maintainability
    * Consistency
    * Readability

---

## Where `allOf` Can Be Used

Although this lecture demonstrates `allOf` under `components/schemas`, it is **not limited to schemas**.

You can use `allOf` in:

* Request bodies
* Responses
* Inline schemas
* Anywhere a schema is allowed

---

## Mental Model to Remember

Think of `allOf` as:

> “Take everything from these schemas and **combine them into one**”

If inheritance is your goal → `allOf` is the correct tool.
