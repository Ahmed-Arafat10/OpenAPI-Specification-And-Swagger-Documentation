## 53. Polymorphism using `oneOf` keyword

In this lecture, the focus is on **achieving polymorphism in OpenAPI** using the `oneOf` keyword.

Unlike inheritance (`allOf`), where schemas are *combined*, `oneOf` enforces **mutual exclusivity**.

---

## What `oneOf` Means

`oneOf` tells OpenAPI:

> ÔÇťThe data **must match exactly one** of the schemas listed here.ÔÇŁ

* Ôťů Valid if **only one schema matches**
* ÔŁî Invalid if **none match**
* ÔŁî Invalid if **more than one matches**

This is how **polymorphism** is expressed in OpenAPI.

---

## Problem Being Solved

You want to design an API where:

* An order **can contain either**

    * a `Mobile`
    * **or** a `Laptop`
* **Not both at the same time**

This is a classic polymorphic request-body scenario.

---

## High-Level Scenario

* A new API is introduced under the **Orders** tag
* Endpoint: `/oneOfOrder`
* HTTP Method: `POST`
* Request body:

    * Requires `products`
    * Requires `address`
* `products`:

    * Must be **either Mobile OR Laptop**
    * Enforced using `oneOf`

---

## Why `oneOf` is Polymorphism

The **shape of the request changes** depending on the scenario:

* Sometimes the request contains a `Mobile`
* Sometimes it contains a `Laptop`

Same API Ôćĺ different valid forms
That is **polymorphism**.

---

## YAML Code ÔÇô `oneOf` in Request Body

Below is the **exact structural idea** used in the lecture, simplified and focused on `oneOf`.

```yaml
paths:
  /oneOfOrder:
    post:
      tags:
        - Orders
      summary: Place order using oneOf
      description: >
        This API accepts an order with either a Mobile or a Laptop,
        but not both.
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required:
                - products
                - address
              properties:
                products:
                  oneOf:
                    - $ref: '#/components/schemas/Mobile'
                    - $ref: '#/components/schemas/Laptop'
                address:
                  $ref: '#/components/schemas/Address'
      responses:
        '200':
          description: Order placed successfully
```

---

## How Validation Works Here

### Ôťů Valid Requests

* Order with **only Mobile**
* Order with **only Laptop**

### ÔŁî Invalid Requests

* Order with **both Mobile and Laptop**
* Order with **neither Mobile nor Laptop**

OpenAPI tools and validators will enforce this automatically.

## Mental Model (Very Important)

* **`allOf`** Ôćĺ *combine schemas*
* **`oneOf`** Ôćĺ *choose exactly one schema*
* **`anyOf`** Ôćĺ *choose one or more schemas* (next lecture)

If your requirement is:

> ÔÇťAccept **only one possible structure at a time**ÔÇŁ
