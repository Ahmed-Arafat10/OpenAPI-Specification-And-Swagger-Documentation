## 34. Introduction to CommonMark Syntax

In this section, the focus shifts from **functional correctness** of the OpenAPI Specification to **documentation quality and readability**. Up to this point, API descriptions have been written as plain text using the `description` fields available across the specification. While this works, it is limited in expressiveness.

This lecture introduces **CommonMark syntax**, which allows you to significantly improve how API documentation is presented—without leaving the OpenAPI ecosystem.

---

### Why CommonMark Is Needed in OpenAPI

Almost every level of an OpenAPI document supports a `description` field:

* API-level descriptions (`info.description`)
* Operation descriptions
* Parameter descriptions
* Request body and response descriptions
* Schema field descriptions

If these descriptions are written as plain text, the documentation:

* Becomes hard to scan
* Lacks visual hierarchy
* Cannot highlight important points

CommonMark solves this by enabling **rich-text formatting** directly inside OpenAPI descriptions.

---

### What Is CommonMark in the Context of OpenAPI

* OpenAPI supports **CommonMark-compliant Markdown** in all `description` fields
* This allows descriptions to include:

    * Headings
    * Bold and italic text
    * Lists
    * Code snippets
    * Links
    * Structured documentation blocks

This means OpenAPI documentation can be as expressive as:

* Word documents
* PowerPoint slides
* GitHub README files

—but still remain **machine-readable** and **standard-compliant**.

---

### Headings in Descriptions

You can create headings using the hash (`#`) symbol:

* One hash → largest heading
* Two hashes → smaller heading
* Three hashes → even smaller heading

This is useful for structuring long descriptions into clear sections.

---

### Text Emphasis

CommonMark supports multiple emphasis styles:

* *Italic text* using single asterisks
* **Bold text** using double asterisks
* ***Bold + italic*** using triple asterisks

This is particularly useful for:

* Highlighting warnings
* Emphasizing constraints
* Drawing attention to important behavior or rules

---

### Lists for Structured Information

You can represent lists using hyphens:

* Ideal for:

    * Feature lists
    * Validation rules
    * Behavioral notes
    * Step-by-step explanations

Lists dramatically improve readability compared to long paragraphs.

---

### Links and External References

CommonMark allows embedding links inside descriptions:

* Useful for:

    * External documentation
    * Terms of service
    * Licensing pages
    * Support portals

These links are rendered as clickable URLs in tools like Swagger UI.

---

### Writing Code Inside Descriptions

One of the most powerful features is the ability to include **code snippets**.

There are two options:

* Inline code (single backtick)

    * Useful for short examples like field names or values
* Code blocks (triple backticks)

    * Useful for:

        * JSON examples
        * Request/response samples
        * Usage examples

The backtick character ( ` ) is located on the same key as the tilde (`~`), just below the Escape key on most keyboards.

---

### Why CommonMark Is Widely Used

CommonMark is not exclusive to OpenAPI. It is widely adopted across the industry because it strikes a balance:

* More expressive than plain text
* Much simpler than HTML and CSS
* Easy to learn
* Easy to maintain

This makes it ideal for technical documentation.

---

### Real-World Industry Usage

CommonMark is heavily used in:

* GitHub README files (`README.md`)
* Technical documentation
* Developer portals
* API reference docs

GitHub repositories rely on CommonMark to render:

* Headings
* Tables
* Images
* Bullet lists
* Code blocks

The same rendering principles apply when Swagger UI displays OpenAPI descriptions written using CommonMark.

---

### How This Applies to OpenAPI Going Forward

From this point onward:

* Descriptions in OpenAPI should **not be treated as plain text**
* They should be treated as **rich documentation blocks**
* CommonMark will be used to:

    * Improve API clarity
    * Reduce misunderstandings
    * Make APIs easier to adopt by consumers

This section lays the foundation for writing **professional-grade API documentation** directly inside OpenAPI specifications, without relying on external documents.

With this understanding, future lectures will apply CommonMark practically inside OpenAPI descriptions.
