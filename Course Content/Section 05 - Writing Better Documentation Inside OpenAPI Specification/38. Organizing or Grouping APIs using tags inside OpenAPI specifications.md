## 38. Organizing or Grouping APIs Using Tags in OpenAPI Specification

As your API surface grows, **discoverability becomes a real problem**. When all endpoints appear under a single **Default** section in Swagger UI, consumers are forced to scroll and search manually, which hurts usability—especially when you have dozens or hundreds of APIs.

OpenAPI solves this problem using **tags**.

Tags allow you to **group related APIs into logical sections** such as:

* Categories
* Products
* Orders

Swagger UI automatically uses these tags to organize endpoints into collapsible sections, making your documentation **clean, navigable, and consumer-friendly**.

---

### Conceptual Overview

Tags work in **two steps**:

1. **Define tag metadata at the root level**

    * Tag name
    * Tag description

2. **Attach tags to operations**

    * Each operation (GET, POST, DELETE, etc.) declares which tag(s) it belongs to

A single API operation:

* Can belong to **one tag**
* Or **multiple tags**, if needed

---

## 1️⃣ Defining Tags at the Root Level

Tags are declared as a **root-level array** in the OpenAPI document.

### YAML Example — Root `tags` Definition

```yaml
tags:
  - name: categories
    description: Category related operations

  - name: products
    description: Product related operations

  - name: orders
    description: Order related operations
```

Key points:

* `tags` is an **array**
* Each tag is an object with:

    * `name` (used for grouping)
    * `description` (shown in Swagger UI)
* Tag names are **case-sensitive**

---

## 2️⃣ Using Tags Inside Path Operations

Once tags are defined at the root, they must be **referenced inside operations**.

The `tags` keyword is added **inside an operation object**, directly under the HTTP method.

---

### YAML Example — Categories API

```yaml
paths:
  /categories:
    get:
      tags:
        - categories
      summary: Get all categories
      responses:
        '200':
          description: Successful response
```

This causes the `/categories` GET API to appear under the **Categories** section in Swagger UI.

---

### YAML Example — Products API

```yaml
paths:
  /products:
    get:
      tags:
        - products
      summary: Get all products
      responses:
        '200':
          description: Successful response
```

This endpoint will now appear under the **Products** section.

---

### YAML Example — Orders API

```yaml
paths:
  /orders:
    post:
      tags:
        - orders
      summary: Create an order
      responses:
        '201':
          description: Order created successfully
```

---

## 3️⃣ Assigning Multiple Tags to a Single API (Optional)

An operation can belong to **multiple tags** if it logically fits in more than one group.

### YAML Example — Multiple Tags on One Operation

```yaml
paths:
  /categories:
    get:
      tags:
        - categories
        - orders
      summary: Categories usable in orders
      responses:
        '200':
          description: Successful response
```

Result:

* The same API appears under **Categories**
* And also under **Orders**

This is optional and should be used **only when it makes semantic sense**, otherwise it can confuse consumers.

---

## 4️⃣ What Swagger UI Does with Tags

Once tags are configured correctly:

* APIs are grouped into **collapsible sections**
* Each section is labeled using the **tag name**
* Tag descriptions appear as **section headers**
* Default grouping disappears

This dramatically improves:

* Readability
* Navigation
* Consumer experience

---

## Practical Guidelines

* Always define tags **before** using them
* Use **business-oriented names**, not technical ones
* Avoid excessive cross-tagging unless necessary
* Keep tag names **short and meaningful**
* Group APIs the way **consumers think**, not how backend code is structured

---

## Why Tags Matter in Real Projects

In real-world APIs:

* Teams grow
* APIs grow
* Consumers vary in expertise

Tags turn your OpenAPI document from a **flat API list** into a **well-organized API catalog**, which is critical for:

* Public APIs
* Partner APIs
* Internal developer portals

With tags, your documentation scales **cleanly and professionally** as your API ecosystem grows.
