## 35. Writing Better Documentation Using CommonMark Syntax

At this stage, our OpenAPI Specification already contains correct structural elements. However, **correctness alone is not enough**—documentation quality plays a major role in how easily consumers understand and adopt your APIs.

Until now, the `description` field has been written as plain text. While this works functionally, it results in **flat, monotonous documentation** that does not encourage readers to explore further. This lecture focuses on transforming those plain descriptions into **clear, structured, and engaging documentation** using **CommonMark syntax**.

---

### Improving the API-Level Description (`info.description`)

The `info.description` field is the **first thing users read** when viewing your API in tools like Swagger UI. This makes it the ideal place to:

* Explain the purpose of the API
* Set expectations for consumers
* Highlight key features and benefits

Instead of a single paragraph, CommonMark allows this section to be structured like professional documentation.

---

### Using Headings for Structure

Headings create a clear hierarchy and improve readability.

* A single `#` creates a large heading
* `##` and `###` create smaller subheadings

Example use cases:

* “About Us”
* “Supported Categories”
* “How This API Helps You”

This makes the documentation easy to scan and visually organized.

---

### Emphasizing Important Text

CommonMark allows emphasis without clutter:

* **Bold text** for key concepts
* *Italic text* for emphasis
* ***Bold + italic*** for critical highlights

This is especially useful when:

* Introducing the product or service name
* Emphasizing API-related terms
* Drawing attention to important concepts

---

### Highlighting Keywords with Inline Code Style

Using inline code formatting (single backticks) is a powerful way to:

* Highlight domain-specific terms
* Draw attention to important keywords
* Make certain phrases stand out visually

This style is ideal for:

* Business terms (e.g., commissions, roles)
* API-related terminology
* Feature names

It creates visual contrast without overwhelming the reader.

---

### Presenting Lists for Clarity

Lists are far more readable than paragraphs when describing multiple items.

Using hyphens, you can:

* List supported categories
* Show feature sets
* Break down offerings clearly

Nested lists can also be used to represent subcategories or grouped data, which is especially useful for product-based APIs.

---

### Optional: Code Blocks for Examples

CommonMark supports multi-line code blocks using triple backticks.

These are useful when:

* Showing sample requests or responses
* Demonstrating example payloads
* Explaining expected data formats

Even if you don’t need them immediately, knowing how to use them gives you flexibility for future documentation needs.

---

### Why This Matters

Well-written documentation:

* Reduces back-and-forth communication
* Minimizes misunderstandings
* Improves API adoption
* Makes your API feel professional and reliable

Since OpenAPI descriptions are rendered directly in tools like Swagger UI, **every improvement you make here directly improves the developer experience**.

---

### Practical Guidance

* Use CommonMark primarily inside `description` fields
* Focus on clarity, not decoration
* Highlight what matters most to API consumers
* Structure content so it can be skimmed quickly

With these techniques, your OpenAPI documentation becomes more than a technical artifact—it becomes a **developer-friendly guide** that explains not just *how* to use your APIs, but *why* they matter.
