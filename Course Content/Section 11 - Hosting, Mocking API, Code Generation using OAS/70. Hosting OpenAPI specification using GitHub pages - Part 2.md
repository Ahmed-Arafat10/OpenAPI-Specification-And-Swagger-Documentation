## 70. Hosting OpenAPI Specification Using GitHub Pages – Part 2

This lecture shows **how to actually host and publish** an OpenAPI specification as a **live website** using **GitHub Pages**, so consumers can always access the **latest Swagger UI view** of your APIs.

---

## Goal of This Lecture

* Convert an OpenAPI YAML file into a **live documentation website**
* Use **Swagger UI** to render the specification
* Host everything using **GitHub Pages**
* Eliminate manual sharing of YAML files with consumers

---

## Hosting Options Overview

OpenAPI documentation can be hosted using many platforms, such as:

* GitHub Pages
* AWS S3
* GitLab Pages
* Dropbox
* Google Drive
* Internal web servers or application servers

The **only technical requirement** is:

* An **HTML file**
* An **OpenAPI YAML file**

With these two files, Swagger UI can render your documentation anywhere.

This lecture focuses on **GitHub Pages** because it is:

* Free
* Developer-friendly
* Widely used in the developer community

---

## The Core Challenge

At this point, we only have:

* An **OpenAPI YAML file**

But a website requires:

* HTML
* CSS
* JavaScript

So the question becomes:

> How do we convert an OpenAPI YAML file into a website?

---

## Solution: Swagger UI HTML Wrapper

Swagger UI already provides:

* Prebuilt CSS
* Prebuilt JavaScript

We just need a **simple HTML file** that:

* Loads Swagger UI assets
* Points Swagger UI to our OpenAPI YAML file

---

## Using the Course GitHub Repository

Inside the course repository, there is a folder named:

```
openapi-deployment
```

This folder contains:

* `index.html`
* The OpenAPI YAML file

These two files are all we need to deploy our documentation.

---

## Understanding `index.html`

The HTML file is intentionally simple.

### Key Sections

#### 1. Import Swagger UI CSS

Using a CDN hosted by Swagger UI:

* Loads all styling needed for Swagger UI
* No local CSS files required

#### 2. Import Swagger UI JavaScript

Using a CDN:

* Loads `SwaggerUIBundle`
* This JavaScript library renders the OpenAPI spec

#### 3. Initialize Swagger UI

Inside `window.onload`:

* A `SwaggerUIBundle` object is created
* The `url` points to the OpenAPI YAML file
* `dom_id` defines where Swagger UI should render

#### 4. DOM Container

Inside `<body>`:

* A `<div>` with the same `id` as `dom_id`
* Swagger UI renders inside this container

---

## Important Configuration Detail

The YAML file must be:

* In the **same folder** as `index.html`
* Or referenced via a **valid URL**

The filename can be anything (e.g., `openapi.yaml`), as long as:

* It contains a valid OpenAPI specification
* The path is correct in `index.html`

---

## Creating a GitHub Pages Repository

To host a GitHub Pages website, **two rules must be followed**:

### Rule 1: Repository Naming Convention

The repository name must be:

```
<github-username>.github.io
```

Example:

```
easybytes.github.io
```

### Rule 2: Repository Must Be Public

GitHub Pages works only with:

* Public repositories (for free hosting)

---

## Uploading the Files

Once the repository is created:

1. Upload:

    * `index.html`
    * `openapi.yaml`
2. Commit the changes to the repository

That’s it—no build tools, no servers, no configuration files.

---

## GitHub Pages Deployment

After committing:

* GitHub automatically starts **Pages build & deployment**
* This can be monitored in the **Actions** tab
* Once the status turns green:

    * The site is live

---

## Accessing the Hosted Documentation

The documentation becomes available at:

```
https://<github-username>.github.io
```

At this URL:

* Swagger UI loads automatically
* All APIs are rendered
* Schemas, paths, security, and examples are visible
* Consumers can explore and test APIs

---

## Downloading the YAML from the Website

Swagger UI provides a **Download** option:

* Consumers can download the exact YAML file
* Ensures transparency
* Confirms what version is being viewed

---

## How the “Magic” Works

* `index.html` loads Swagger UI assets
* Swagger UI fetches the OpenAPI YAML
* Swagger UI renders the full documentation dynamically

This is functionally equivalent to Swagger Editor—**but hosted publicly**.

---

## Benefits of This Approach

* Single URL for all consumers
* Always up to date
* No manual YAML sharing
* No need for consumers to run Swagger UI locally
* Easy updates:

    * Modify YAML
    * Push to GitHub
    * Website updates automatically

## Practical Recommendation

Even if you don’t use GitHub Pages:

* Use the **same pattern**
* Deploy `index.html` + OpenAPI YAML
* Host them on any platform you prefer

This is the **recommended professional approach** for publishing OpenAPI documentation.

````html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <link rel="stylesheet" type="text/css" href="//unpkg.com/swagger-ui-dist@3/swagger-ui.css">

  <title>FarmStall API v1</title>

<body>

  <div id="eazyshop-oas" />

  <script src="//unpkg.com/swagger-ui-dist@3/swagger-ui-bundle.js"></script>
  <script>

    window.onload = function() {

      const ui = SwaggerUIBundle({
        url: "openapi.yml",
        dom_id: "#eazyshop-oas",
        deepLinking: true,
      })

    }

  </script>

</body>
</html>
````
