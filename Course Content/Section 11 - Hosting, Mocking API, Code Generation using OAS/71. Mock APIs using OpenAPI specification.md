## 71. Mock APIs Using OpenAPI Specification

This lecture introduces **API mocking** using an OpenAPI document and demonstrates how consumers and producers can **work in parallel** by simulating APIs **before backend development is complete**.

---

## Why Mock APIs Matter

In real projects:

* Backend APIs may take time to implement
* Frontend or client teams still need endpoints to integrate against
* Waiting for producers blocks progress

With **OpenAPI-based mocking**:

* Consumers can start development immediately
* Producers and consumers work **in parallel**
* Integration issues are discovered early
* Final integration becomes smoother

This is one of the **biggest practical advantages** of having a well-defined OpenAPI specification.

---

## Core Idea Behind Mocking with OpenAPI

If your OpenAPI specification includes:

* Correct paths
* Request/response schemas
* **Example response data**

Then a mock server can:

* Pretend to be your real API
* Return example responses
* Enforce paths, methods, and even security

No backend code (Java, Python, etc.) is required.

---

## Mock Server Used in This Lecture

The lecture uses **Prism**:

* Open-source HTTP mock server
* Reads OpenAPI (v3) and Swagger (v2) specs
* Simulates API behavior based on the spec
* Returns example-based responses

Because the course uses **OpenAPI 3**, Prism is a perfect fit.

---

## High-Level Steps to Mock APIs with Prism

There are **three main steps**:

1. Install **Node.js**
2. Install **Prism CLI**
3. Start the mock server using your OpenAPI YAML file

Once these steps are done, your APIs are instantly mockable.

---

## Step 1: Install Node.js

Prism is a Node.js-based tool.

* Download Node.js from the official website
* Install it based on your OS (Windows, macOS, Linux)

### Verify Installation

```bash
node --version
```

If a version number is printed, Node.js is installed successfully.

---

## Step 2: Install Prism CLI

Install Prism globally using npm:

```bash
npm install -g @stoplight/prism-cli
```

### Notes

* On macOS or Linux, you may need:

  ```bash
  sudo npm install -g @stoplight/prism-cli
  ```
* On Windows, run the command prompt as **Administrator**

This installs all Prism-related binaries needed to run mock servers.

---

## Step 3: Start the Mock Server

Navigate to the folder containing your OpenAPI YAML file.

Verify file presence:

```bash
ls
```

Then run:

```bash
prism mock ./openapi.yaml -p 88
```

### Explanation

* `prism mock` → starts Prism in mock mode
* `./openapi.yaml` → path to your OpenAPI specification
* `-p 88` → port number for the mock server

---

## Mock Server Startup Output

Once started, Prism:

* Parses the OpenAPI spec
* Lists all recognized endpoints
* Starts listening on:

```
http://127.0.0.1:88
```

At this point, your **mock APIs are live**.

---

## Testing Mock APIs in the Browser

You can immediately test APIs by visiting endpoints in a browser.

Example:

```
http://localhost:88/categories
```

Prism returns:

* JSON response
* Based on **example data** defined in the OpenAPI spec

No backend exists — the response comes purely from the spec.

---

## Security Behavior in Mock APIs

If an API is secured in the OpenAPI spec:

* Prism enforces that security
* Browser or client will prompt for authentication

Important note:

* Because these are mock APIs:

    * **Any username/password/token is accepted**
    * Security checks are simulated, not validated

This allows consumers to test secured flows without real credentials.

---

## Why Example Data Is Critical

Prism relies heavily on **examples** defined in OpenAPI responses.

If examples are missing:

* Prism has no data to return
* Mock responses may fail or be empty

That’s why:

* Well-defined examples are essential
* Especially for consumer-driven development

---

## Switching Between Multiple Examples

If an API response defines **multiple examples**, Prism:

* Uses the **first example by default**

But you can override this using the `Prefer` header.

### Example: Selecting a Specific Example

```http
Prefer: example=singleCategory
```

This tells Prism:

* Which example to return
* Without changing the OpenAPI spec

---

## Switching Response Status Codes

Prism also allows switching **entire responses**, such as testing error cases.

### Example: Force a 500 Error

```http
Prefer: code=500
```

Result:

* Prism returns the `500` response
* Based on the OpenAPI definition

This is extremely useful for:

* Testing error handling
* Simulating failures
* Frontend robustness testing

---

## Testing with Postman

In Postman:

1. Call the mock API URL
2. Add authentication (if required)
3. Add `Prefer` headers to:

    * Select examples
    * Switch response codes
4. Send the request

Postman behaves exactly as if it were calling a real backend.

---

## Parallel Development Workflow

With OpenAPI + Prism:

* Consumers:

    * Build client code
    * Test integrations
    * Handle edge cases
* Producers:

    * Implement real APIs independently

When backend is ready:

* Consumers switch:

    * Mock URL → Real API URL
* No surprises
* Contract already validated

---

## Key Takeaways

* OpenAPI enables **contract-first development**
* Prism allows instant API mocking
* Examples drive mock responses
* Security rules are respected (but not validated)
* `Prefer` header allows:

    * Example switching
    * Error simulation
* Consumers no longer need to wait for producers
