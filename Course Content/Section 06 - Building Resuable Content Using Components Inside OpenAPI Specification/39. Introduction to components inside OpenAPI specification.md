## 39. Introduction to `components` inside OpenAPI Specification

So far, we have discussed most of the **root-level objects** supported by OpenAPI:

* `openapi`
* `info`
* `servers`
* `paths`
* `tags`

The remaining important root objects are:

* `components`
* `security`
* `externalDocs`

In this lecture, the focus is **only on `components`**, because it plays a **critical role in writing clean, maintainable, and scalable OpenAPI specifications**.

---

## Why `components` Are Needed

If you look at a real OpenAPI document (like the one built so far in the course), you will quickly notice **duplication**:

* The same **query parameter** (e.g. `orderId`) is defined in:

    * `GET /orders`
    * `DELETE /orders`
* The same **schema structures** (e.g. Product, Category, Order) are repeated in:

    * Request bodies
    * Responses
* The same **response definitions** (e.g. 200, 404, 500) appear again and again

This leads to several problems:

* The specification becomes **long and hard to read**
* Any change requires updating **multiple places**
* High chance of **inconsistencies and bugs**
* Poor maintainability as APIs evolve

This problem is **exactly the same problem solved by functions/classes in programming languages**.

---

## What `components` Solve

The `components` object allows you to:

* Define **reusable pieces** once
* Reference them anywhere using `$ref`
* Avoid duplication
* Enforce consistency
* Make changes in **one place only**

`components` act as a **central registry** of reusable definitions.

---

## What Can Be Defined Inside `components`

Inside the `components` root object, OpenAPI allows defining reusable definitions such as:

* `schemas` – reusable data models
* `parameters` – reusable query/path/header parameters
* `responses` – reusable response objects
* `requestBodies` – reusable request body definitions
* `headers`
* `securitySchemes`

(In this lecture, the focus is on **schemas, parameters, request bodies, and responses**.)

---

## High-Level Structure of `components`

```yaml
components:
  schemas:
    ...
  parameters:
    ...
  responses:
    ...
  requestBodies:
    ...
```

Each section can contain **multiple named definitions**, which can later be referenced.

---

## YAML Illustration – Defining a Reusable Schema

### Define a schema inside `components`

```yaml
components:
  schemas:
    Address:
      type: object
      properties:
        addressLine:
          type: string
        city:
          type: string
        state:
          type: string
        zipCode:
          type: string
```

Here:

* `Address` is the schema name
* It is defined **once**
* It can now be reused anywhere in the OpenAPI document

---

## YAML Illustration – Referencing a Component Using `$ref`

### Using the reusable schema in a response

```yaml
responses:
  '200':
    description: Successful response
    content:
      application/json:
        schema:
          $ref: '#/components/schemas/Address'
```

Key points about `$ref`:

* Always starts with `#/`
* `#/components/schemas/Address` means:

    * Look inside `components`
    * Go to `schemas`
    * Use the schema named `Address`

---

## YAML Illustration – Reusable Parameters

### Define a reusable parameter

```yaml
components:
  parameters:
    OrderIdParam:
      name: orderId
      in: query
      required: true
      schema:
        type: integer
```

### Use it in multiple operations

```yaml
paths:
  /orders:
    get:
      parameters:
        - $ref: '#/components/parameters/OrderIdParam'
      responses:
        '200':
          description: Order fetched

    delete:
      parameters:
        - $ref: '#/components/parameters/OrderIdParam'
      responses:
        '204':
          description: Order deleted
```

Now:

* The parameter is defined **once**
* Used in **multiple APIs**
* Any change happens in a **single location**

---

## YAML Illustration – Reusable Request Body

```yaml
components:
  requestBodies:
    ProductRequest:
      required: true
      content:
        application/json:
          schema:
            $ref: '#/components/schemas/Product'
```

Usage:

```yaml
paths:
  /products:
    post:
      requestBody:
        $ref: '#/components/requestBodies/ProductRequest'
      responses:
        '201':
          description: Product created
```

---

## YAML Illustration – Reusable Responses

```yaml
components:
  responses:
    NotFoundResponse:
      description: Resource not found
      content:
        application/json:
          schema:
            type: string
```

Usage:

```yaml
responses:
  '404':
    $ref: '#/components/responses/NotFoundResponse'
```

---

## Key Takeaways About `components`

* `components` is a **root-level object**
* It is **optional**, but **essential for real-world APIs**
* Promotes:

    * Reusability
    * Consistency
    * Maintainability
* Uses `$ref` to reference definitions
* Works like:

    * Functions
    * Classes
    * Constants
      in programming languages

---

## Mental Model to Remember

> **If you copy-paste something more than once → it belongs in `components`.**

With this mindset, your OpenAPI specifications will remain:

* Clean
* Scalable
* Easy to evolve
* Professional-grade
