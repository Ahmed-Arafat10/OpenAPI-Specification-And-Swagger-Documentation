## 3. Why `paths` Is the Most Important Section

The `paths` object is where **actual APIs are defined**.

Everything before this point answers questions like:

* Who owns the API?
* What is the API about?
* Where is it hosted?

The `paths` object answers:

* **What APIs exist**
* **What endpoints are available**
* **How consumers interact with them**

This is where OpenAPI moves from **documentation metadata** to **API behavior definition**.

---

## 4. High-Level Structure of `paths`

`paths` is a **root-level object**.

Inside `paths`:

* Each key starts with `/`
* Each key represents **one API endpoint**

Example conceptually:

```text
paths
 ├─ /endpoint1
 ├─ /endpoint2
 └─ /endpoint3
```

Each endpoint is called a **Path Item Object**.

---

## 5. Path Item Object (Endpoint Level)

For each path (for example `/products`), you define:

* Which HTTP methods are supported:

    * GET
    * POST
    * PUT
    * DELETE
    * etc.

Each HTTP method is defined **under the same path**.

Conceptually:

```text
/products
 ├─ get
 ├─ post
 └─ delete
```

This allows:

* One URL
* Multiple operations
* Different behaviors per HTTP method

---

## 6. Operation Object (HTTP Method Level)

Under each HTTP method, you define an **Operation Object**.

This object describes **what the API operation does**.

Typical information inside an operation includes:

* What the operation does
* What request it expects
* What responses it returns

Even though this may look complex now, it will be broken down step by step in upcoming lectures.

---

## 7. Operation Metadata

Inside each operation, you can describe:

* A short explanation of the operation
* A detailed explanation of what happens
* A unique identifier for the operation

This helps:

* Consumers understand intent
* Tools generate meaningful method names
* Documentation stay clear and readable

---

## 8. Request Handling in Paths

Most APIs accept input.

This input is described using:

* A **request body**
* Optional parameters (path, query, header — covered later)

The request body definition explains:

* What data the consumer must send
* The structure of that data
* The format of that data

---

## 9. Response Handling in Paths

Every API operation can return **multiple responses**.

Examples:

* Success response
* Authentication failure
* Server error

Because of this:

* Responses are defined **per HTTP status code**
* Each response can have:

    * Its own description
    * Its own response structure

This allows precise documentation of **all possible outcomes**.

---

## 10. Response Object Structure (High Level)

For each response:

* A description explains what the response means
* Content defines how the response body looks

Different status codes can return:

* Different data structures
* Different meanings
* Different schemas

This makes the API contract explicit and predictable.

---

## 11. Big Picture Hierarchy Recap

At a very high level, the structure looks like this:

```text
openapi
info
servers
paths
 └─ /path
    └─ http-method
       └─ operation
          ├─ request
          └─ responses
```

This hierarchy may look heavy at first glance, but:

* Each level has a clear responsibility
* Each piece will be explained in isolation
* Examples will make it intuitive
