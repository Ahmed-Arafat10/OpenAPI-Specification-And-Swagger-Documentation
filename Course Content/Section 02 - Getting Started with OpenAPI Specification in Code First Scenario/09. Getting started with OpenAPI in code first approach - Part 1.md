## 2. Two Common Scenarios When Adopting OpenAPI

Whenever you decide to introduce OpenAPI Specification in a project, you will almost always fall into **one of two scenarios**.

---

### 2.1 Scenario 1: APIs Already Exist (Code-First)

In many organizations:

* APIs were developed **years ago**
* APIs were released to production **without OpenAPI**
* Documentation was either:

    * Missing
    * Incomplete
    * Manually written

After release:

* Consumers start asking many questions
* Integration issues increase
* Documentation becomes a bottleneck

At this point, the organization realizes the **importance of OpenAPI Specification**.

This scenario is called the **Code-First approach**.

---

### 2.2 Scenario 2: APIs Are Not Built Yet (Design-First)

In newer or well-structured projects:

* API requirements are discussed first
* Design documents are created
* API contracts are finalized
* Only then does coding start

Here:

* OpenAPI is written **before implementation**
* API design drives development

This is called the **Design-First approach**.

> This lecture focuses **only on the Code-First approach**.

---

## 3. What Is the Code-First Approach?

In the **Code-First approach**:

* APIs already exist
* They may already be deployed to production
* You want to **document them using OpenAPI**

Key point:

* You **do not** start writing OpenAPI from scratch
* You leverage tooling to generate an initial specification

This saves:

* Time
* Manual effort
* Guesswork

---

## 4. Bootstrapping OpenAPI Using Swagger Tools

Instead of writing OpenAPI files line by line, the instructor introduces a **faster approach**.

This approach uses:

* **SwaggerHub**
* **SwaggerHub Explorer**

These tools are maintained by **SmartBear**.

---

## 5. How SwaggerHub Explorer Helps in Code-First

### 5.1 Core Idea

SwaggerHub Explorer can:

* Send requests to existing API endpoints
* Observe request and response structures
* Analyze API behavior automatically

Based on this analysis:

* It generates an **OpenAPI Specification document**
* This document becomes the **initial version** of your API contract

---

### 5.2 What Happens Behind the Scenes

1. You provide API URLs
2. Requests are sent to the APIs
3. Responses are captured
4. Request/response schemas are inferred
5. An OpenAPI document is generated

This OpenAPI spec:

* Describes endpoints
* Includes request structures
* Includes response structures
* Covers most validation rules

---

## 6. Value of the Generated OpenAPI Specification

In many real-world cases:

* The generated OpenAPI document is **good enough**
* It already covers:

    * Request fields
    * Response fields
    * Data types
    * Basic validations

Organizations can:

* Use it immediately
* Improve or refine it later if needed

This drastically lowers the barrier to OpenAPI adoption for legacy systems.

---

## 7. Role of SwaggerHub in the Workflow

Once the OpenAPI specification is generated:

* It can be stored in **SwaggerHub**
* Teams can:

    * Collaborate
    * Version the specification
    * Improve documentation
    * Share APIs with consumers

SwaggerHub acts as:

* A **central repository** for API specifications
* A collaboration platform for API teams

---

## 8. Why Code-First Is Common in the Industry

The instructor emphasizes that:

* Many organizations did **not start with OpenAPI**
* APIs were built first, standards came later
* Code-First adoption is therefore very common

SwaggerHub Explorer provides a **practical bridge** between:

* Legacy APIs
* Modern API standards

## 10. Additional Important Note

**(Added by me – not mentioned in the lecture)**

> In Code-First projects, the generated OpenAPI specification should always be **reviewed and refined manually**, because automated tools may miss business rules, edge cases, or authentication details that are not visible from raw request/response traffic.
