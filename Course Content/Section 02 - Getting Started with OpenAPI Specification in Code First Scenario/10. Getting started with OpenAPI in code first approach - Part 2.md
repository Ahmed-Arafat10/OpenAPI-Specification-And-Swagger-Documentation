# 10. Getting Started with OpenAPI in Code-First Approach – Part 2

## 1. Purpose of This Lecture

This lecture is the **hands-on continuation** of the Code-First approach.

The objective here is to:

* Use **SwaggerHub Explorer**
* Test **existing APIs**
* Prepare the **API interaction history**
* Lay the groundwork for **generating an OpenAPI Specification automatically**

At this stage, we are **not generating the OpenAPI document yet**—we are preparing the data required to generate it in the next lecture.

---

## 2. Tool Used: SwaggerHub Explorer

From the Swagger website:

* Navigate to **Tools**
* Select **SwaggerHub Explorer**

This tool allows you to:

* Test REST APIs
* Capture request & response data
* Convert that interaction history into an OpenAPI Specification

---

## 3. Important Update: Swagger Inspector → SwaggerHub Explorer

There is an important naming and lifecycle change highlighted in this lecture.

### 3.1 Deprecation Notice

* **Swagger Inspector** is **deprecated**
* It was officially **sunset on September 29, 2023**
* It has been **replaced** by **SwaggerHub Explorer**

SwaggerHub Explorer:

* Is the modern replacement
* Has UI improvements
* Offers a more powerful API exploration experience

### Key Clarification

If you hear someone mention:

* *Swagger Inspector*

They are effectively referring to:

* **SwaggerHub Explorer**

This distinction is important for:

* Project discussions
* Reading older documentation
* Understanding legacy references

## 5. Supported Protocols & Operations

SwaggerHub Explorer currently supports:

* **REST APIs**
* **Kafka-based APIs**

For REST APIs, you can select:

* HTTP methods:

    * GET
    * POST
    * PUT
    * DELETE
    * PATCH
    * OPTIONS
* Server URL
* API path

You then send requests and inspect responses.

## 6. Challenge: No APIs to Test

At this point, a practical problem arises:

> We don’t have any APIs built specifically for this course.

Building APIs from scratch:

* Is time-consuming
* Is language-dependent (Java, Python, Node.js, etc.)
* Is **not the focus** of this course

So instead:

* The lecture uses **publicly available APIs**
* These APIs return **mock or test data**
* They are perfect for demonstration and learning

---

## 7. Public API Option #1: Star Wars API (SWAPI)

A public API based on the **Star Wars universe** is used.

Capabilities:

* Fetch people data
* Fetch planets
* Fetch starships
* Fetch vehicles and more

### 7.1 Base URL Usage

* The base URL from the website is copied
* Pasted into **SwaggerHub Explorer**
* Server and path are auto-detected

---

### 7.2 Testing a GET API – People

Example path:

* `/people/1`

Steps:

1. Select protocol: **REST**
2. Select operation: **GET**
3. Paste the path
4. Click **Send**

Response includes:

* Name
* Height
* Mass
* Hair color
* Skin color
* Eye color
* Birth year
* Gender
* Homeworld
* Films

## 8. Public API Option #2: ReqRes (Request–Response API)

A second public API is introduced as a **backup option**, in case the first site is unavailable.

This API supports:

* GET
* POST
* PUT
* DELETE

---

### 8.1 Testing a GET API – List Users

Steps:

1. Copy base URL
2. Paste into SwaggerHub Explorer
3. Add path (e.g. `/api/users`)
4. Select **GET**
5. Click **Send**

Response:

* List of users
* User IDs
* Names
* Emails
* Avatars

---

### 8.2 Testing a POST API – Create User

Steps:

1. Select **POST**
2. Use path: `/api/users`
3. Paste request body (JSON)
4. Ensure JSON format is selected
5. Click **Send**

Response:

* HTTP **201**
* User created
* Generated user ID

---

### 8.3 Testing a PUT API – Update User

Steps:

1. Select **PUT**
2. Use path: `/api/users/2`
3. Paste request body
4. Click **Send**

Response:

* HTTP **200**
* User updated
* Timestamp returned

---

## 9. Best Practices When Using SwaggerHub Explorer

Very important guidance from the lecture:

* **Do NOT use production URLs**
* Avoid exposing:

    * Sensitive data
    * Real customer information
* Always use:

    * Dev environment
    * Staging environment
    * Mock/test APIs

SwaggerHub Explorer sends real requests—treat it accordingly.

---

## 10. API History in SwaggerHub Explorer

After testing APIs:

* Navigate to **History**
* You should see:

    * All tested APIs
    * Server URLs
    * Paths
    * HTTP methods
    * Status codes

Important notes:

* History may take **1–2 minutes** to populate
* Ensure all required APIs appear before proceeding

This history is **critical**, because:

* It will be used to generate the OpenAPI Specification in the next lecture

## 12. Additional Important Note

**(Added by me – not mentioned in the lecture)**

> Automatically generated OpenAPI specs from API traffic usually lack business-level constraints (authentication rules, rate limits, edge-case validations). Treat the generated document as a **baseline**, not a final contract, and refine it manually afterward.
