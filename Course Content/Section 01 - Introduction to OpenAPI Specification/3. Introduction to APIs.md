# 3. Introduction to APIs

## 1. Purpose of This Lecture

Before diving into **OpenAPI Specification** and **Swagger tools**, this lecture establishes a **clear and practical understanding of APIs themselves**.
The goal is to ensure you fully understand:

* What an API is
* How APIs enable communication between software systems
* Real-world scenarios where APIs are used extensively

This foundation is critical, because **OpenAPI is not about building APIs**, but about **describing them correctly**.

---

## 2. What Is an API?

**API** stands for **Application Programming Interface**.

An API defines the **allowed interaction between two pieces of software**.

These software components can be:

* **Local** (running on the same machine)
* **Remote** (running on different servers or locations)

At its core, an API specifies:

* What operations (methods) are allowed
* What inputs are accepted (parameters / request structure)
* What outputs are returned (response structure)

All these details together form what we call an **API definition**.

---

## 3. What Does an API Contain?

An API definition typically includes:

* **List of operations / methods**
  What actions can be performed

* **Input parameters**
  What data must be sent to invoke an operation

* **Request structure**
  How the request payload should look

* **Response structure**
  How the response data will be returned

This definition acts as a **contract** between interacting software systems.

---

## 4. Local APIs vs Remote APIs

### 4.1 Local APIs

Local APIs enable interaction between software components **within the same system**.

#### Example: Operating System & Applications

* Your operating system
* Applications installed on your laptop (browser, document editor, etc.)

When you:

* Click an icon
* Open a file
* Perform an action on the UI

The operating system communicates with applications through **local APIs**.

This interaction happens entirely within the same machine.

---

### 4.2 Remote APIs

Remote APIs enable communication between **different applications deployed on different servers or locations**, typically over a network.

In remote API communication, there are two key roles:

* **Provider / Producer**

    * The system that exposes services via an API

* **Consumer**

    * The system that calls or consumes those services

Between the provider and consumer, an **API contract** is established:

* Request format
* Response format
* Expected behavior

If this contract changes without coordination, **communication breaks**.

---

## 5. API Contract Concept

An API contract defines:

* How requests must be sent
* How responses will be returned
* What data structures are valid

Both provider and consumer must strictly follow this contract.

If:

* Schemas change
* Request/response formats change unexpectedly

Then:

* API communication fails
* Systems stop working together

This concept of **contract stability** is a key reason why **API documentation and specifications are critical**.

---

## 6. How Are APIs Usually Built?

Most APIs in modern organizations are built using:

* **HTTP**
* **REST architectural style**
* **TCP/IP networking standards**

These technologies are widely used because they are:

* Lightweight
* Easy to build
* Easy to deploy
* Platform-independent

As a result, **the majority of real-world APIs you encounter will be HTTP + REST based**.

---

## 7. Real-World API Examples

### 7.1 E-Commerce & Payment APIs

Consider an e-commerce platform like **Amazon**.

Typical flow:

1. User adds items to cart
2. User proceeds to checkout
3. User selects a payment option:

    * Credit card
    * **PayPal**
    * Bank transfer

Behind the scenes:

* Amazon and PayPal are **two different systems**
* They belong to **different organizations**
* They communicate via APIs

Roles:

* **PayPal** → Provider (exposes payment APIs)
* **E-commerce application** → Consumer

When the user clicks “Pay with PayPal”:

* An API call is made
* Payment is processed
* Response is returned

All of this happens **without the user seeing any of it**.

---

### 7.2 Travel Booking & Airline APIs

Consider a travel booking website.

Instead of checking:

* Airline A website
* Airline B website
* Airline C website

The user:

* Enters source
* Enters destination
* Selects travel date
* Chooses number of passengers

Behind the scenes:

* The travel site calls APIs from multiple airline systems
* Examples include airlines like:

    * **Etihad Airways**
    * **Singapore Airlines**

Roles:

* **Airline systems** → Providers
* **Travel booking website** → Consumer

The booking website:

* Aggregates responses
* Displays price and flight options to the user

From the user’s perspective:

* It looks like one system
  But internally:
* **Dozens of API calls may be happening simultaneously**

---

## 8. APIs Are Everywhere

APIs are not limited to:

* E-commerce
* Travel systems

They are used in:

* Mobile applications → backend servers
* Web applications → backend services
* IoT devices → cloud platforms
* Internal microservices → other microservices

APIs are **invisible to end users**, yet they power almost everything.

---

## 9. APIs as “Unsung Heroes”

APIs are often called **unsung heroes** because:

* Users never see them
* Developers rely on them constantly
* Entire systems depend on their reliability

Behind every UI action:

* A click
* A swipe
* A form submission

There is usually **one or more API calls** happening in the background.
