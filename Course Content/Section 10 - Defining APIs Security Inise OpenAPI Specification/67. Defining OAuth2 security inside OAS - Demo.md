## 67. Defining OAuth2 Security Inside OAS – Demo

This lecture demonstrates **how to configure OAuth2 security** inside an OpenAPI document and **how Swagger UI exposes OAuth2 authorization options** once it is defined.

---

## Goal of This Demo

* Define an **OAuth2 security scheme** under `components.securitySchemes`
* Configure a specific **OAuth2 flow**
* Define **scopes**
* Apply OAuth2 security using the `security` keyword
* Observe how Swagger UI handles OAuth2 authorization

---

## Step 1: Add OAuth2 Security Scheme

OAuth2 security schemes are defined under the `components` object, alongside schemas, parameters, and other reusable definitions.

### Location

At the end of the OpenAPI document:

```
components:
```

---

### OAuth2 Security Scheme Definition

A new security scheme is created with an arbitrary name.

```yaml
components:
  securitySchemes:
    oauth2Auth:
      type: oauth2
      flows:
        authorizationCode:
          authorizationUrl: https://auth.example.com/oauth/authorize
          tokenUrl: https://auth.example.com/oauth/token
          scopes:
            read: Read access to resources
            write: Write access to resources
            admin: Admin-level access
```

---

### Explanation of Fields

* **`oauth2Auth`**

    * Arbitrary name
    * Must be unique
    * Used later in the `security` keyword

* **`type: oauth2`**

    * Mandatory for OAuth2

* **`flows`**

    * OAuth2 supports multiple flows
    * In this demo, **authorizationCode** is used

* **`authorizationUrl`**

    * Endpoint where users authenticate and give consent

* **`tokenUrl`**

    * Endpoint where access tokens are issued

* **`scopes`**

    * Defines permission levels
    * In this demo:

        * `read` → read-only access
        * `write` → read and write access
        * `admin` → elevated privileges

Scope names are **completely customizable** and depend on backend authorization rules.

---

## Step 2: Apply OAuth2 Security Using `security`

After defining the scheme, OAuth2 must be **enforced**.

### Root-Level Security Configuration

At the top of the document (above `paths`):

```yaml
security:
  - oauth2Auth:
      - read
      - write
      - admin
```

---

### Important Difference from Other Mechanisms

* OAuth2 **does NOT use empty brackets**
* Scopes **must be explicitly listed**
* These scopes represent what the API requires or supports

---

## Step 3: Swagger UI Behavior

Once OAuth2 is defined and applied:

* Swagger UI shows **OAuth2** as an authorization option
* Clicking **Authorize** displays:

    * Client ID
    * Client Secret
    * Available scopes (read, write, admin)

Consumers can:

* Select which scopes they want to request
* Authenticate against the authorization server
* Receive an access token (if allowed)

---

## Understanding Scope Enforcement

Scopes are not just documentation—they reflect **real backend rules**.

Example:

* If a client is registered for `read` only:

    * Requesting `admin` will fail
* If a client requests allowed scopes:

    * Authorization succeeds
    * Access token is issued

Swagger UI simply mirrors what the **OAuth2 authorization server** enforces.

---

## Resulting Security Options in Swagger UI

At this point, Swagger UI lists **all configured security mechanisms**, including:

* Basic Authentication
* Bearer Authentication
* API Key
* OAuth2 (Authorization Code flow)

This makes the API’s security model **fully transparent** to consumers.

---

## Key Takeaways

* OAuth2 security schemes are defined using:

    * `type: oauth2`
    * `flows`
* Authorization Code flow is:

    * Most common
    * Most secure
* Scopes are **mandatory**
* OAuth2 security is applied using the `security` keyword
* Swagger UI:

    * Prompts for client credentials
    * Allows scope selection
    * Executes real OAuth2 flows
