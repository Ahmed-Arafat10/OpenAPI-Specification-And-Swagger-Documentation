## 63. Defining HTTP Basic & Bearer Authentication Inside OAS – Theory

This lecture explains the **theoretical foundation** of two commonly used authentication mechanisms—**HTTP Basic Authentication** and **Bearer Authentication**—and how both are **described (not enforced)** inside the **OpenAPI Specification**.

---

## Goal of This Lecture

* Understand **how Basic and Bearer authentication work**
* Learn **how to describe these mechanisms in OpenAPI**
* Know **where security definitions live** and **how they are applied**
* Prepare for **hands-on implementation** in the next lecture

---

## HTTP Basic Authentication

### How Basic Authentication Works

* The client sends credentials with **every request**
* Credentials are sent via an HTTP header:

```
Authorization: Basic <base64(username:password)>
```

### Example

If:

* Username: `easybytes`
* Password: `admin123`

Then:

* Raw value: `easybytes:admin123`
* Encoded using Base64
* Sent as:

```
Authorization: Basic ZWFzeWJ5dGVzOmFkbWluMTIz
```

### Important Characteristics

* Credentials are **Base64 encoded**, not encrypted
* Credentials are sent **on every request**
* Simple but **less secure** unless combined with HTTPS

---

## Describing Basic Authentication in OpenAPI

### Step 1: Define a Security Scheme

Security schemes are defined under:

```
components → securitySchemes
```

Example (conceptual):

```yaml
components:
  securitySchemes:
    basicAuth:
      type: http
      scheme: basic
```

Key points:

* `basicAuth` is an **arbitrary name**
* `type` must be `http`
* `scheme` must be `basic`

---

### Step 2: Apply the Security Scheme Using `security`

The `security` object **enforces** the scheme.

#### Root-Level Security (Applies to All APIs)

```yaml
security:
  - basicAuth: []
```

Important rule:

* **Square brackets are mandatory**
* For Basic authentication, the array is **empty**

Why empty?

* Scopes are only applicable to advanced mechanisms like OAuth2 or OpenID
* Basic authentication does **not support scopes**

---

### Operation-Level Security (Optional)

Instead of applying security globally, it can be applied **per operation**:

```yaml
paths:
  /categories:
    get:
      security:
        - basicAuth: []
```

This allows:

* Some APIs to be secured
* Others to remain public

---

## Bearer Authentication

### How Bearer Authentication Works

* Clients do **not send username and password**
* Instead, they send a **token** (commonly a JWT)
* Token is sent via the same header:

```
Authorization: Bearer <token>
```

### Why Bearer Authentication Is Preferred

* Username and password are usually sent **only once**
* Backend generates a token (e.g., JWT)
* Token is reused for subsequent requests
* Reduces exposure of credentials
* More secure and scalable

---

## Describing Bearer Authentication in OpenAPI

### Security Scheme Definition

```yaml
components:
  securitySchemes:
    bearerAuth:
      type: http
      scheme: bearer
      bearerFormat: JWT
```

Key differences from Basic authentication:

* `scheme` is `bearer`
* `bearerFormat` describes the token format (e.g., `JWT`)

---

### Applying Bearer Authentication

```yaml
security:
  - bearerAuth: []
```

Notes:

* Still uses **square brackets**
* Still **no scopes**
* Application rules are identical to Basic authentication

---

## Scope Clarification (Very Important)

* Scopes are used only with:

    * OAuth2
    * OpenID Connect
* Basic and Bearer authentication:

    * **Do not support scopes**
    * Always use empty arrays (`[]`)

---

## Root-Level vs Operation-Level Security

You have **full flexibility**:

### Root-Level

* Applies security to **all APIs**
* Best for fully secured systems

### Operation-Level

* Applies security selectively
* Useful for:

    * Public + private APIs
    * Health-check endpoints
    * Login or token-generation APIs

---

## Key Takeaways

* Basic Authentication:

    * Sends Base64-encoded username/password
    * Simple but less secure
* Bearer Authentication:

    * Sends token (e.g., JWT)
    * More secure and widely used
* OpenAPI requires:

    * `components.securitySchemes` → definition
    * `security` → enforcement
* Square brackets are **mandatory**
* Scopes are **not applicable** for these mechanisms
