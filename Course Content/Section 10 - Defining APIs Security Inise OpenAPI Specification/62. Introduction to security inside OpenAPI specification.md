## 62. Introduction to Security Inside OpenAPI Specification

This lecture introduces **API security concepts** and explains **how security is described in the OpenAPI Specification** at a high level before implementation begins.

---

## Why API Security Is Mandatory

In real-world projects:

* APIs are **never exposed publicly without security**
* Not every consumer is allowed to access your APIs
* Unsecured APIs can lead to:

    * Data breaches
    * Unauthorized access
    * Severe security vulnerabilities

Any consumer who wants to interact with your APIs must:

* Understand the **security requirements**
* Send the required authentication details with each request

Because of this, **security is a first-class concern** in API design.

---

## Security Support in the OpenAPI Specification

OpenAPI provides **built-in support for describing API security requirements** in a standardized way.

It allows you to document **how clients should authenticate** when calling your APIs.

---

## Common Security Mechanisms Supported by OpenAPI

OpenAPI supports multiple widely-used authentication and authorization standards, including:

* **HTTP Basic Authentication**
* **Bearer Authentication**
* **API Key–based Authentication**
* **OAuth 2.0**
* **OpenID Connect**

These mechanisms are commonly used across:

* APIs
* Microservices
* Inter-service communication
* Third-party integrations

In real projects, one or more of these mechanisms are always enforced.

---

## Why These Security Mechanisms Matter

Consider scenarios such as:

* One microservice calling another
* A frontend application consuming backend APIs
* External partners accessing your platform APIs

Without security:

* Anyone could consume your services
* Sensitive data could be exposed
* Abuse and attacks become inevitable

That is why security must be **explicitly defined and enforced**.

---

## How Security Is Described in OpenAPI

To describe API security in OpenAPI, **two key objects are required**:

---

### 1. `securitySchemes`

* Defined under the **`components` object**
* Similar to how we define:

    * Schemas
    * Responses
    * Parameters
    * Request bodies
* This is where you **define authentication mechanisms**

Examples of what you define here:

* Type of authentication (Basic, Bearer, API Key, OAuth2, OpenID)
* Required headers or tokens
* Authorization flow details

---

### 2. `security`

* Used to **apply security requirements to APIs**
* Refers to one or more security schemes defined under `components.securitySchemes`
* Can be applied at:

    * Root level (applies to all APIs)
    * Operation level (applies to specific APIs)

This is how security is **enforced** after being defined.

---

## Relationship Between `securitySchemes` and `security`

The process works in two steps:

1. **Define security mechanisms**

    * Using `securitySchemes` under `components`
2. **Apply those mechanisms**

    * Using the `security` keyword

Both are required:

* Defining a scheme alone does not enforce it
* Using `security` without a scheme is invalid

---

## High-Level Summary of the Security Flow

* OpenAPI supports multiple standard security approaches
* Security is defined using:

    * `securitySchemes` (definition)
    * `security` (application)
* APIs without security are unrealistic in production systems
* This section will focus heavily on security due to its importance
