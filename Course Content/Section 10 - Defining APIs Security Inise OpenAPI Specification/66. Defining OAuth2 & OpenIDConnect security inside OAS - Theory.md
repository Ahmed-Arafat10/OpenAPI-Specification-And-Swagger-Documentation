## 66. Defining OAuth2 & OpenID Connect Security Inside OAS – Theory

This lecture explains **OAuth 2.0** and **OpenID Connect** from an **OpenAPI documentation perspective**.
The goal is **not** to master these frameworks, but to understand **how to describe their security requirements** inside the **OpenAPI Specification**.

---

## Why OAuth2 & OpenID Connect Are Important

In enterprise-grade applications:

* Simple mechanisms (Basic, Bearer, API Key) are often **not enough**
* APIs require:

    * Fine-grained authorization
    * User consent
    * Limited access to user data
* OAuth2 and OpenID Connect are:

    * Mature
    * Widely adopted
    * Industry standards

Most large organizations rely on these frameworks.

---

## High-Level Idea of OAuth2

OAuth2 allows a **client application** to access **user data** from a **resource server** *without* knowing the user’s credentials.

### Real-World Example

* A third-party app wants to analyze:

    * Twitter posts
    * Facebook activity
    * LinkedIn data
* The user **should NOT** give their username/password to that app

### What Happens Instead

1. User is redirected to:

    * Twitter / Facebook / LinkedIn login page
2. User logs in **directly** with the provider
3. User gives **consent** to share specific data
4. Provider issues an **access token**
5. Third-party app uses the access token to fetch allowed data

---

## Resource Owner & Consent

* **Resource owner**: the end user
* The user controls:

    * Whether access is granted
    * What data is shared
* Credentials are **never exposed** to the client application

---

## What Are Scopes

Scopes define **what the client is allowed to do**.

Examples:

* `profile` → access basic profile info
* `posts` → access posts only
* `read` → read-only access
* `write` → create/update data

Scopes prevent over-permission:

* Client gets **only what is allowed**
* Nothing more

---

## OAuth2 Flows (Grant Types)

OAuth2 supports **multiple flows**, each designed for a specific scenario.

Common flows include:

* **authorizationCode**

    * End user involved
    * Most secure and widely used
* **clientCredentials**

    * System-to-system communication
    * No user involved
* **password**

    * Legacy (not recommended today)
* **implicit**

    * Older browser-based flow (largely deprecated)

Which flow to use depends on:

* Whether a user is involved
* Type of client (browser, backend, machine)

---

## Describing OAuth2 in OpenAPI

### Security Scheme Definition

To describe OAuth2, you must define a **security scheme** with:

* `type: oauth2`
* One or more **flows**

Example (conceptual):

```yaml
components:
  securitySchemes:
    oauth2Auth:
      type: oauth2
      flows:
        authorizationCode:
          authorizationUrl: https://auth.example.com/authorize
          tokenUrl: https://auth.example.com/token
          scopes:
            read: Read access
            write: Write access
```

### Key Points

* Flow name must match a valid OAuth2 flow
* URLs describe where:

    * Authorization happens
    * Tokens are issued
* Scopes are **mandatory** for OAuth2

---

## Applying OAuth2 Security

Unlike Basic or Bearer authentication:

* OAuth2 **requires scopes**
* Empty square brackets (`[]`) are **not allowed**

Example:

```yaml
security:
  - oauth2Auth:
      - read
      - write
```

This means:

* Client must request and provide these scopes
* Access is limited accordingly

---

## OpenID Connect (OIDC)

OpenID Connect is **built on top of OAuth2**.

### Key Difference from OAuth2

* OAuth2 → **access token only**
* OpenID Connect → **access token + ID token**

---

## What Is an ID Token

An **ID token** contains **user identity information**, such as:

* Email
* Username
* User ID
* Contact details

It is usually:

* A **JWT**
* Issued by the identity provider

This allows client applications to:

* Know *who* the user is
* Not just *what* they can access

---

## Why OpenID Connect Is Preferred Today

* Everything OAuth2 provides
* Plus identity information
* Standardized user authentication
* Widely adopted across enterprises

Most modern systems use **OpenID Connect instead of plain OAuth2**.

---

## Describing OpenID Connect in OpenAPI

### Security Scheme Definition

```yaml
components:
  securitySchemes:
    openIdAuth:
      type: openIdConnect
      openIdConnectUrl: https://auth.example.com/.well-known/openid-configuration
```

### Important Notes

* `type` must be `openIdConnect`
* `openIdConnectUrl` is mandatory
* This URL exposes:

    * Authorization endpoints
    * Token endpoints
    * Supported scopes

---

## Applying OpenID Connect Security

Just like OAuth2, scopes must be specified:

```yaml
security:
  - openIdAuth:
      - profile
      - email
```

Scopes control:

* What identity data is shared
* What access level is granted

---

## Key Differences Summary

| Aspect          | OAuth2 | OpenID Connect |
| --------------- | ------ | -------------- |
| Access Token    | ✅      | ✅              |
| ID Token        | ❌      | ✅              |
| User Identity   | ❌      | ✅              |
| Built On OAuth2 | ❌      | ✅              |

---

## Key Takeaways

* OAuth2 & OpenID Connect are **advanced security frameworks**
* Used heavily in enterprise systems
* OAuth2 focuses on **authorization**
* OpenID Connect adds **authentication**
* Both require:

    * Security schemes under `components`
    * Explicit scopes under `security`
* Scopes are **mandatory**, not optional
