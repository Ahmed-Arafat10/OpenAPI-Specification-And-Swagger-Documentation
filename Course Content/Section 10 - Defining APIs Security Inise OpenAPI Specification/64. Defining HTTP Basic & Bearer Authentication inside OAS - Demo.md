## 64. Defining HTTP Basic & Bearer Authentication Inside OAS – Demo

This lecture demonstrates **how to implement HTTP Basic Authentication and Bearer Authentication** inside an OpenAPI document and **how Swagger UI behaves once security is applied**.

---

## Objective of This Demo

* Define **security schemes** in the OpenAPI specification
* Apply those schemes using the `security` keyword
* Observe how Swagger UI:

    * Marks APIs as secured
    * Collects authentication details
    * Sends credentials automatically in requests

---

## Step 1: Defining Security Schemes

Security schemes are defined under the **`components` object**, alongside schemas, parameters, responses, and links.

### Location

At the end of the OpenAPI document, under:

```
components:
```

### Security Schemes Definition

Two security schemes are defined:

1. **Basic Authentication**
2. **Bearer Authentication**

#### Example Structure

```yaml
components:
  securitySchemes:
    basicAuth:
      type: http
      scheme: basic

    bearerAuth:
      type: http
      scheme: bearer
      bearerFormat: JWT
```

### Important Notes

* `basicAuth` and `bearerAuth` are **arbitrary names**
* They must be **unique** within `securitySchemes`
* These names are later referenced inside the `security` keyword
* `bearerFormat` is used to describe the token format

    * `JWT` (JSON Web Token) is the most commonly used example

---

## Step 2: Applying Security Using the `security` Keyword

After defining the security schemes, they must be **applied**.

### Root-Level Security Configuration

Security is applied near the top of the file, just above the `paths` object.

```yaml
security:
  - basicAuth: []
  - bearerAuth: []
```

### What This Means

* Both authentication mechanisms are supported
* Square brackets (`[]`) are mandatory
* The arrays are empty because:

    * Basic and Bearer authentication **do not support scopes**
* Applying security at the root level means:

    * **All APIs are secured**

---

## Result in Swagger UI

Once the security configuration is added:

### Visual Indicators

* An **Authorize** button appears in Swagger UI
* Every API operation shows a **lock icon**
* This indicates:

    * APIs are protected
    * Authentication is required

---

## Understanding the Authorization Options

Clicking the **Authorize** button shows:

1. **Basic Authentication**

    * Accepts:

        * Username
        * Password

2. **Bearer Authentication**

    * Accepts:

        * Token (e.g., JWT)

This makes it clear to consumers:

* Which authentication mechanisms are supported
* What information they must provide

---

## Executing Requests with Authentication

### Basic Authentication Flow in Swagger UI

1. Enter username and password
2. Click **Authorize**
3. Swagger UI stores the credentials
4. When executing an API:

    * Swagger automatically adds the header:

```
Authorization: Basic <base64-encoded-credentials>
```

This allows consumers to:

* Test secured APIs directly
* Avoid manually crafting headers

---

## Lock vs Unlock Icons Explained

### Locked Icon (🔒)

* Authentication details are present
* Requests include authorization headers

### Unlocked Icon (🔓)

* No authentication details provided
* Requests are sent without authorization headers

Clicking **Logout** clears saved credentials and switches icons back to unlocked.

---

## Flexibility in Security Configuration

* You are **not required** to use both authentication methods
* You can:

    * Use only Basic authentication
    * Use only Bearer authentication
    * Use a combination
* The OpenAPI specification must always reflect:

    * What your backend actually supports

Mismatch between backend security and OpenAPI documentation leads to confusion.

---

## Key Takeaways

* Security schemes are defined under `components.securitySchemes`
* Security is enforced using the `security` keyword
* Root-level security secures all APIs
* Swagger UI automatically:

    * Displays authorization options
    * Injects headers into requests
* Empty scope arrays (`[]`) are mandatory for Basic and Bearer authentication

---

This concludes the **practical demo** of defining HTTP Basic and Bearer Authentication inside the **OpenAPI Specification**.
