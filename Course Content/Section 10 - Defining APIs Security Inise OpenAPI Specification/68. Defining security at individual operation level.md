## 68. Defining Security at Individual Operation Level

In previous lectures, security was defined **at the root level** of the **OpenAPI Specification**.
Because of that, **all API operations automatically inherited those security requirements**.

This lecture explains how to **override or customize security at the individual operation level**.

---

## Root-Level Security Recap

When security is defined at the root level:

```yaml
security:
  - basicAuth: []
  - bearerAuth: []
  - apiKeyAuth: []
  - oauth2Auth:
      - read
      - write
```

* Every API operation is protected
* Swagger UI shows a **lock icon** on all endpoints
* Consumers must provide authentication details for every API

---

## Why Operation-Level Security Is Needed

Real-world APIs often have mixed requirements:

* Some APIs require **strong authentication**
* Some APIs require **simple authentication**
* Some APIs must be **public** (no authentication)

OpenAPI allows you to handle all these cases cleanly.

---

## Overriding Security at Operation Level

Security can be defined **inside an individual operation object** (`get`, `post`, etc.).

When defined at operation level:

* It **overrides** root-level security
* Root-level security is ignored for that operation

---

## Scenario 1: Restrict an API to a Single Security Mechanism

### Requirement

For `GET /categories`, allow **only Basic Authentication**, even though multiple security mechanisms exist globally.

### Implementation

Inside the operation object:

```yaml
paths:
  /categories:
    get:
      operationId: getCategories
      security:
        - basicAuth: []
```

### Result

* This API supports **only Basic Authentication**
* Other mechanisms (Bearer, API Key, OAuth2) are ignored
* Swagger UI shows **only Basic Auth** when clicking the lock icon for this API

---

## Scenario 2: Make an API Public (No Security)

### Requirement

For `GET /categories/{categoryId}`, allow **no authentication at all**.

### Implementation

```yaml
paths:
  /categories/{categoryId}:
    get:
      operationId: getCategoryDetails
      security: []
```

### What `security: []` Means

* Completely disables security for this operation
* Overrides root-level security
* No credentials are required

### Result in Swagger UI

* No lock icon is displayed
* API is publicly accessible
* Anyone can call it without headers or tokens

---

## Important Rule to Remember

* **Root-level security** → default for all APIs
* **Operation-level security** → overrides root-level security
* `security: []` → explicitly means **no security**

This behavior is intentional and very powerful.

---

## Mixing Security Strategies

You can freely mix and match:

* Root-level security for most APIs
* Operation-level overrides for:

    * Public endpoints
    * Login/token endpoints
    * Health checks
    * Special business rules

This makes OpenAPI suitable for **real production systems**.
