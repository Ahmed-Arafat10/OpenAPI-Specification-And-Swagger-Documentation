## 65. Defining API Key Security Inside OAS

This lecture introduces **API Key–based security** and explains how to **describe and apply it** inside the **OpenAPI Specification**.

---

## What Is API Key Authentication

API Key authentication is a security mechanism where:

* The client sends a **pre-generated key** (token-like value)
* The key identifies and authorizes the consumer
* The backend validates the key before processing the request

Unlike Basic or Bearer authentication:

* No username/password is sent
* No JWT parsing is required
* The API key is usually issued by the API provider

---

## Where API Keys Can Be Sent

OpenAPI allows API keys to be sent in **three possible locations**:

1. **Request Header**
2. **Query Parameter**
3. **Cookie**

The API provider decides **one** of these, and consumers must follow it exactly.

---

## Describing API Key Security in OpenAPI

Describing API key security follows the **same structure** as previous security mechanisms, with a few key differences.

---

## Step 1: Define API Key Security Scheme

API key authentication is defined under:

```
components → securitySchemes
```

### Key Differences from Basic & Bearer

* `type` must be `apiKey`
* You must specify:

    * **Where** the key is expected (`in`)
    * **What name** it uses (`name`)

---

### Example Security Scheme Definition

```yaml
components:
  securitySchemes:
    apiKeyAuth:
      type: apiKey
      in: header
      name: X-API-KEY
```

---

## Explanation of Fields

### `type`

* Must be `apiKey`
* Case-sensitive (should be written exactly as shown)

### `in`

Specifies **where the API key is sent**:

* `header`
* `query`
* `cookie`

### `name`

* Represents:

    * Header name (if `in: header`)
    * Query parameter name (if `in: query`)
    * Cookie name (if `in: cookie`)
* In this lecture, the chosen name is:

    * `X-API-KEY`

---

## Step 2: Apply API Key Security Using `security`

Once defined, the API key scheme must be enforced.

### Root-Level Security Configuration

```yaml
security:
  - apiKeyAuth: []
```

---

## Important Notes About Scopes

* API key authentication **does not support scopes**
* The empty array (`[]`) is mandatory
* Scopes are only used with:

    * OAuth2
    * OpenID Connect

---

## Swagger UI Behavior

After applying API key security:

* Swagger UI shows **API Key** as an authorization option
* Consumers can:

    * Enter the API key value
    * Click **Authorize**
* Swagger UI automatically:

    * Adds the API key to requests
    * Uses the exact name and location defined in the spec

---

## Request Execution Example

When executing an API after authorization:

* Swagger UI injects a header like:

```
X-API-KEY: <your-api-key-value>
```

This confirms:

* The OpenAPI definition is correctly interpreted
* Consumers know exactly how to authenticate

---

## What API Consumers Must Do

When calling the API from real clients (Java, Python, JavaScript, etc.):

* Send the API key:

    * In the same location (`header`, `query`, or `cookie`)
    * With the exact same name
* Use the key provided by the API owner

Mismatch in name or location will result in authentication failure.

---

## Flexibility and Best Practices

* API Key security can be:

    * Used alone
    * Combined with other security mechanisms
* Always document **exactly what your backend expects**
* Never assume consumers know:

    * Header names
    * Token locations
    * Authentication rules

OpenAPI exists to eliminate this ambiguity.

---

## Key Takeaways

* API Key authentication is simple and widely used
* Defined using:

    * `type: apiKey`
    * `in`
    * `name`
* Applied using the `security` keyword
* Does not support scopes
* Swagger UI automatically injects API keys into requests
