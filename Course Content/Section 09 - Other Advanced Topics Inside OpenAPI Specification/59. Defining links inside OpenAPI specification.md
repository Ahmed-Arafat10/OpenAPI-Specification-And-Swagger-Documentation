## 59. Defining Links Inside OpenAPI Specification

This lecture introduces **Links**, a feature added in **OpenAPI 3.0**, and explains how links help describe **relationships between API operations**.

---

## What Are Links in OpenAPI

Links allow you to describe **how values returned by one API operation response can be used as input for another API operation**.

This is especially useful in systems where:

* APIs are **interrelated**
* One API produces data that another API consumes
* Consumers need to understand **API flow**, not just isolated endpoints

Links are defined as part of the **response object**.

---

## Why Links Were Introduced

Before links:

* API consumers had to manually figure out:

    * Which API to call next
    * Where a required ID or value comes from

With links:

* The OpenAPI document itself explains:

    * API relationships
    * Data flow between operations

This improves:

* API discoverability
* Developer experience
* Documentation clarity

---

## Conceptual Example (User APIs)

A typical example described in the lecture:

1. **Create User API**

    * `POST /users`
    * Returns `201 Created`
    * Response body contains a newly generated `id`

2. **Get User API**

    * `GET /users/{userId}`
    * Requires `userId` as input

There is a **direct relationship**:

* The `id` returned by the first API
* Is required as input for the second API

This relationship can be explicitly defined using **links**.

---

## Where Links Are Defined

Links are defined:

* **Inside the response object** of an API operation
* Under a field named `links`
* Optionally reusable via `components`

---

## Core Elements of a Link

A link definition includes:

* **Unique link name**
* **Description**
* **operationId**

    * The `operationId` of the target operation
* **Parameters mapping**

    * How response values map to request parameters of the next API

---

## Mapping Response Data to Request Parameters

When defining a link:

* The **target API** may expect parameters
* Those parameters can be mapped from:

    * Response body
    * Response headers
    * Other response fields

### Response-to-Request Mapping Syntax

```yaml
$response.body#/fieldName
```

This means:

* Take `fieldName` from the response body
* Pass it as an input to the linked operation

---

## Example Mapping Explanation

* Response body contains: `id`
* Target API parameter name: `userId`

Mapping example:

```yaml
userId: $response.body#/id
```

This explicitly tells consumers:

* Where `userId` comes from
* Which API produced it

---

## Applying Links to the Course Example (Orders APIs)

In the course OpenAPI document, a similar scenario exists:

1. **Create Order**

    * `POST /orders`
    * Returns a response containing `orderId`

2. **Get Order Details**

    * `GET /orders`
    * Accepts `orderId` as a query parameter

The response of the first API can be linked to the second API using links.

---

## Defining Links Under Components

Just like schemas, parameters, and responses, **links can be defined under `components`**.

Example structure:

```yaml
components:
  links:
    OrderIdToGetOrder:
      description: Order ID returned in response can be used to fetch order details
      operationId: getOrderDetails
      parameters:
        orderId: $response.body#/orderId
```

Key points:

* The link name (`OrderIdToGetOrder`) must be unique within `components.links`
* `operationId` must match an existing operation
* Parameter names must exactly match the target API parameter names

---

## Important Naming Rules

When defining links:

* Parameter name inside `links.parameters`:

    * Must match the **request parameter name** of the target API
* Response field name:

    * Must exist in the response body schema
* Case and spelling must match exactly

Otherwise, the link will be invalid or misleading.

---

## Referencing Links Inside Responses

Once the link is defined in `components`, it must be **referenced inside the response object**.

Example:

```yaml
responses:
  '201':
    description: Order created successfully
    content:
      application/json:
        schema:
          $ref: '#/components/schemas/OrderResponse'
    links:
      OrderIdToGetOrder:
        $ref: '#/components/links/OrderIdToGetOrder'
```

---

## Result in Swagger UI

After defining the link:

* Swagger UI no longer shows **“No links”**
* The response displays:

    * Link name
    * Description
    * Target operation
    * Parameter mapping details

This visually communicates:

* Which API can be called next
* How to call it
* What data to use

---

## Best Practice for Defining Links

* Prefer defining links inside `components`
* Reuse them using `$ref`
* Keep naming meaningful and descriptive
* Ensure `operationId` is always defined for target operations

This keeps the specification:

* Clean
* Reusable
* Easy to maintain

---

## Why Links Are Valuable for API Consumers

Links provide:

* Clear API flow without external explanation
* Self-documenting API relationships
* Reduced dependency on verbal or written explanations

This acts as **implicit knowledge transfer** from API designers to consumers.

---

## Key Takeaways

* Links were introduced in **OpenAPI 3.0**
* They describe **relationships between operations**
* Links map:

    * Response data → Request parameters
* Defined:

    * Inside response objects
    * Optionally under `components`
* Require:

    * `operationId`
    * Accurate parameter mapping

This concludes the concept and usage of **Links** in the OpenAPI Specification.
