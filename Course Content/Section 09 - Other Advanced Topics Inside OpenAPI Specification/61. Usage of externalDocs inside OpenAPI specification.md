## 61. Usage of `externalDocs` Inside OpenAPI Specification

This lecture introduces the **`externalDocs` object**, another optional but useful feature provided by the **OpenAPI Specification**.

---

## Why `externalDocs` Exists

In many real-world scenarios:

* Your OpenAPI document already contains sufficient technical details
* However, some consumers may want:

    * More business-level explanations
    * Product documentation
    * Company-level API guides
    * Extended usage scenarios

Not all of this information fits naturally inside an OpenAPI specification.

To support these needs, OpenAPI provides the **`externalDocs` object**, which allows you to link to **external resources**.

---

## What Is `externalDocs`

`externalDocs` is an object that allows you to:

* Provide a **URL** to external documentation
* Optionally describe what the external resource contains

This helps consumers navigate beyond the OpenAPI document when needed.

---

## Fields Supported by `externalDocs`

The `externalDocs` object supports **exactly two fields**:

### 1. `description` (Optional)

* A human-readable description
* Explains what the external link contains
* Displayed as the clickable text in Swagger UI

### 2. `url` (Mandatory)

* The external resource URL
* Must be a valid URL
* Without this field, `externalDocs` is invalid

Example structure:

```yaml
externalDocs:
  description: External documentation
  url: https://example.com/docs
```

---

## Where `externalDocs` Can Be Defined

`externalDocs` can be defined at **multiple levels** in the OpenAPI specification:

### 1. Root Level

* Most common and recommended usage
* Applies to the entire API or product
* Typically sufficient for most use cases

### 2. Operation Level

* Can be defined inside individual HTTP method operations
* Uses the same fields: `description` and `url`
* According to the lecture, this is **rarely used in practice**

The instructor recommends defining `externalDocs` **at the root level** in most cases.

---

## Root-Level Usage Example

Placing `externalDocs` just after the `info` object:

```yaml
openapi: 3.0.3
info:
  title: Orders API
  version: 1.0.0

externalDocs:
  description: External documentation
  url: https://example.com/external-docs
```

---

## How It Appears in Swagger UI

* Swagger UI renders a **hyperlink**
* The clickable text comes from the `description` field
* The link points to the provided `url`

Because of this:

* The `description` should be **clear and meaningful**
* Consumers should immediately understand:

    * What kind of information the link provides
    * Why they should click it

---

## Operation-Level Usage (Optional)

Just like the root level, `externalDocs` can also be defined inside:

* Paths
* Individual operations (GET, POST, etc.)

Structure and behavior remain the same.

However:

* This is less commonly used
* Root-level external documentation is usually enough

---

## Practical Guidance from the Lecture

* Use `externalDocs` when:

    * Detailed explanations do not belong inside OpenAPI
    * You already maintain separate documentation portals
* Keep the description meaningful
* Prefer root-level definition unless there is a strong reason not to
