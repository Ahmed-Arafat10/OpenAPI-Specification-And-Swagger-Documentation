## 57. Declaring `operationId` Inside OpenAPI Specification

In the previous lecture, while discussing the **Operation Object** under the Paths section, we identified a field named **`operationId`**.
This lecture focuses entirely on **what `operationId` is**, **why it matters**, and **how to define it correctly** in an OpenAPI document.

---

## What Is `operationId`

* `operationId` is an **optional field** defined inside an **operation object** (i.e., inside an HTTP method such as `get`, `post`, `put`, etc.).
* It represents a **unique string identifier** for a specific API operation.
* The value of `operationId` **must be unique across the entire OpenAPI specification document**.

Even though it is optional, it plays a very important role in real-world usage of OpenAPI.

---

## Why `operationId` Is Important

There are **two major advantages** of defining `operationId`.

---

### 1. Used by Code Generator Tools

One of the major benefits of OpenAPI is that it allows you to generate:

* Client code
* Server-side code

using automated **code generation tools**.

When these tools generate code:

* Each API operation becomes a **method or function** in the generated classes.
* If `operationId` is defined:

    * The code generator uses it as the **method/function name**
* If `operationId` is not defined:

    * The tool may generate **random or non-intuitive method names**

Defining meaningful `operationId` values ensures:

* Clean
* Readable
* Predictable method names in generated code

---

### 2. Required for Linking Operations

Another important advantage of `operationId` is that it enables **linking between operations**.

Example scenario:

* The response of one API becomes the input/request for another API
* You want to explicitly define this relationship inside your OpenAPI specification

Such **operation links** can only be established when:

* Each operation has a unique `operationId`

The detailed concept of **links** will be covered in a **separate lecture**, so full understanding is not required at this point.

---

## What Is Considered an Operation

* Every HTTP method (`get`, `post`, `put`, `delete`, etc.) under a path is considered a **separate operation**
* A single path can:

    * Have **one operation**
    * Or **multiple operations**, depending on how many HTTP methods it supports

Examples:

* `/categories` → one `get` operation
* `/orders` → `get`, `post`, `put`, `delete` operations

Each of these operations can (and should) have its own `operationId`.

---

## Where to Define `operationId`

* `operationId` is defined **inside an HTTP method object**
* It can be placed anywhere inside the method configuration
* Common practice is to place it:

    * After `summary` or `description`

Example structure:

```yaml
paths:
  /categories:
    get:
      summary: Get all categories
      description: Returns all categories from Easy Shop
      operationId: getCategories
```

The position is flexible, but the meaning remains the same.

---

## Naming Strategy for `operationId`

When defining `operationId`, the name should:

* Clearly reflect the **functionality of the API**
* Be **descriptive**
* Be **unique across the entire specification**

Examples of good naming:

* `getCategories`
* `getCategoryDetails`
* `getProducts`
* `getProductDetails`
* `createOrder`
* `updateOrderDetails`
* `deleteOrder`
* `getOrderDetails`

---

## Practical Examples from the Specification

### Categories APIs

* `GET /categories`

    * `operationId`: `getCategories`
    * Returns all categories

* `GET /categories/{categoryId}`

    * `operationId`: `getCategoryDetails`
    * Returns details of a single category

---

### Products APIs

* `GET /products`

    * `operationId`: `getProducts`
    * Returns list of products

* `GET /products/{productId}`

    * `operationId`: `getProductDetails`
    * Returns details of a single product

---

### Orders APIs (Multiple Operations on Same Path)

For paths supporting multiple HTTP methods:

* `POST /orders`

    * `operationId`: `createOrderDetails` (or `saveOrderDetails`)

* `PUT /orders`

    * `operationId`: `updateOrderDetails`

* `DELETE /orders`

    * `operationId`: `deleteOrderDetails`

* `GET /orders`

    * `operationId`: `getOrderDetails`

Each HTTP method is treated as a **separate operation**, even if the path is the same.

---

### Demo APIs (`oneOf` / `anyOf`)

Even demo or example APIs should define `operationId`:

* `GET /oneOfOrder`

    * `operationId`: `demoOfOneOf`

* `GET /anyOfOrder`

    * `operationId`: `demoOfAnyOf`

This maintains consistency and completeness in the specification.

---

## Key Rules to Remember

* `operationId`:

    * Is optional
    * But highly recommended
* Must be:

    * Unique across the entire OpenAPI document
* Used for:

    * Code generation
    * Operation linking
* Should:

    * Match the intent and behavior of the API
