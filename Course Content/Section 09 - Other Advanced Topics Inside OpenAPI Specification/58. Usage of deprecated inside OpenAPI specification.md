## 58. Usage of `deprecated` Inside OpenAPI Specification

In this lecture, the focus is on understanding the **purpose and usage of the `deprecated` field** inside the **Operation Object** in the OpenAPI Specification.

---

## What Is the `deprecated` Field

* `deprecated` is an **optional boolean field**
* It is defined **inside an operation object** (i.e., inside an HTTP method such as `get`, `post`, etc.)
* Example:

```yaml
deprecated: true
```

When this field is set to `true`, the operation is marked as **deprecated**.

---

## How Deprecated Operations Appear

When an operation is marked as deprecated:

* Swagger UI visually highlights the operation
* The operation appears:

    * In a **greyed-out style**
    * With a **strikethrough**
* A visible **warning message** is shown indicating that the operation is deprecated

This makes it immediately clear to API consumers that the operation should not be used going forward.

---

## Why Not Just Delete the Operation?

A common question is:

> If an API is no longer needed, why not simply delete it?

The answer lies in **backward compatibility** and **consumer communication**.

### Real-World Scenarios

There are cases where:

* Existing consumers are still using an API
* Immediate removal would **break client applications**

Instead of deleting the API suddenly, marking it as deprecated allows you to:

* Warn consumers in advance
* Give them time to migrate to a newer API
* Plan a safe removal in future versions

---

## Analogy with Programming Languages

This concept is very similar to how **deprecated classes or methods** work in programming languages:

* Languages like Java or Python do not immediately remove APIs when upgrading versions
* Deprecated methods remain available:

    * For backward compatibility
    * For a few future versions
* Developers are warned not to use them anymore

APIs follow the **same evolution strategy**.

---

## When to Use `deprecated`

You should use `deprecated: true` when:

* You plan to remove an API in future versions
* You want to discourage new consumers from using it
* You want to notify existing consumers to migrate

It acts as an **official deprecation notice**.

---

## Practical Example: Deprecating an API Operation

### Step 1: Introduce a New API Path

A new path is created:

```
/orders/{orderId}
```

* `orderId` is accepted as a **path parameter**
* The operation is similar to an existing `GET /orders` API

---

### Step 2: Define a New Path Parameter

A new parameter is added under `components`:

* Name: `orderId`
* Location: `path`
* Required: `true`
* Schema type: `integer`

This parameter is then referenced in the operation.

---

### Step 3: Provide a Unique `operationId`

Since `operationId` must be unique across the specification, a new value is defined, such as:

```yaml
operationId: getOrderDetailsUsingPathParam
```

---

### Step 4: Mark the Operation as Deprecated

Inside the operation object:

```yaml
deprecated: true
```

As soon as this is applied:

* The entire API operation is visually marked as deprecated in Swagger UI
* Consumers see a warning message when they view or expand the operation

---

## Multiple APIs Serving the Same Purpose

At this stage, there are **two ways to fetch order details**:

1. Using **query parameters**
2. Using **path parameters**

If the design decision is:

* To continue supporting only the query parameter–based API
* And stop supporting the path parameter–based API

Then the correct approach is:

* Keep both APIs temporarily
* Mark the path-parameter version as deprecated

---

## Key Takeaways

* `deprecated` is an **operation-level field**
* It is optional but extremely useful
* It helps:

    * Maintain backward compatibility
    * Communicate API lifecycle changes
* Deprecated APIs:

    * Still function
    * But should not be used by consumers
* Visual warnings in Swagger UI make deprecation obvious

---

This concludes the usage and purpose of the `deprecated` field inside the OpenAPI Operation Object.
